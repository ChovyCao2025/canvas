# 2000+ Execution 并发目标专项 Plan

## 1. 目标

围绕 2000+ execution 并发目标，形成一份分阶段实施路线，先把当前 500~1000 区间能力做实，再进入平台级架构演进。

---

## 2. 实施阶段

## Phase 1：容量基线验证

### 目标
搞清楚当前系统在真实代码和真实配套资源下，500 和 1000 的边界到底在哪里。

### 任务
1. 建立轻链路 / 重链路 / 下游变慢 / 多实例压测场景
2. 补齐监控：
   - execution 并发
   - MySQL 连接池与慢 SQL
   - Redis RT / 连接 / 热点 key
   - RocketMQ backlog / retry backlog
   - 下游 RT / 超时率
3. 验证不同应用集群规模下的真实稳定 execution 并发
4. 输出 500 与 1000 的真实容量基线报告

### 产出
- 容量基线报告
- 当前瓶颈列表
- 基础设施扩容建议

---

## Phase 2：配套资源升级

### 目标
在不大改主架构的情况下，把 1000 区间的配套资源能力补齐。

### 任务
1. MySQL：
   - 升规格
   - 评估主从 / 读写分离
   - 治理热点表
2. Redis：
   - 升规格
   - 准备按执行态 / 缓存态拆分
3. RocketMQ：
   - 评估 broker 吞吐
   - 评估 retry 流量
4. 下游服务：
   - 容量评估
   - 熔断与限流基线

### 产出
- 配套资源升级方案
- 1000 区间稳定运行条件说明

---

## Phase 3：代码治理

### 目标
减少主链路中的同步资源竞争，为 2000+ 做准备。

### 任务
1. 识别热路径中的高频 DB / Redis 往返
2. 识别 execution / request / quota 的写放大问题
3. 治理 retry / replay / overflow 流量
4. 识别并隔离重型任务：
   - audience
   - groovy
   - 大 DAG
5. 梳理下游依赖的连接池与隔离策略

### 产出
- 热路径治理清单
- 代码治理方案
- 重任务隔离方案

---

## Phase 4：架构升级

### 目标
把系统演进成支持 2000+ execution 并发的平台形态。

### 任务
1. 执行通道拆分
   - 轻量实时通道
   - 标准异步通道
   - 重型任务通道
2. 状态分层
   - 强在线状态
   - 弱在线状态
   - 分析/审计状态
3. Redis 分层
   - 执行态 Redis
   - 缓存/路由 Redis
   - bitmap Redis
4. MQ 流量分治
   - 正常流量
   - retry 流量
   - 重型任务流量
5. 下游依赖平台化治理
   - 独立连接池
   - 契约化容量
   - 更严格的熔断限流

### 产出
- 2000+ 平台架构方案
- 实施任务拆解
- 上线与回滚方案

---

## 3. 最终验收

### 验收指标
1. 稳定 execution 并发达到 2000+
2. 高峰期 backlog 可控
3. MySQL / Redis / MQ / 下游指标处于可接受区间
4. 轻任务不被重任务拖垮
5. 架构拆分后容量预测更可控

---

## 4. 建议执行顺序

1. 先做 Phase 1
2. 再做 Phase 2
3. 然后做 Phase 3
4. 最后进入 Phase 4

原因：
- 先确认真实边界
- 再补运维能力
- 再做代码治理
- 最后做架构升级

这样成本最低，也最稳妥。
