# Compliance Data Inventory

Status date: 2026-06-05

This inventory classifies sensitive data families used by the current canvas backend. Each row names the owning context, classification, masking rule, retention rule, and deletion rule so implementation and evidence can be audited against the same source.

| Data family | Fields | Classification | owning context | masking rule | retention rule | deletion rule |
| --- | --- | --- | --- | --- | --- | --- |
| `cdp_user_profile` | `user_id`, `display_name`, `phone`, `email`, `properties_json`, `created_by` | PII / internal | CDP profile | `phone` -> `138****5678`; `email` -> `a***e@example.com`; JSON sensitive keys masked by `PiiMaskingService` | Profile remains online while user is active; inactive profile follows CDP retention policy and legal hold | `DataDeletionService` deletes rows by tenant and user id after dry-run approval |
| `cdp_user_identity` | `identity_type`, `identity_value`, `source_type`, `source_ref_id` | PII / confidential | CDP identity | phone/email/open_id values masked by identity type or key | Same lifecycle as CDP profile; identity history is not retained after approved erasure unless legal hold applies | `DataDeletionService` deletes rows by tenant and user id |
| `cdp_user_tag` and `cdp_user_tag_history` | `tag_code`, `tag_value`, `old_value`, `new_value`, `reason`, `operator` | internal / possible PII | CDP tag operations | Tag values classified per tag definition; PII tags use key-based masking | Current tags follow tag TTL; history retained for operational evidence until retention cutoff | Current `cdp_user_tag` rows are deleted by `DataDeletionService`; history requires retention/legal-hold review before purge |
| `customer_profile` | `timezone`, `region`, `lifecycle_stage`, `attributes` | internal / possible PII | Marketing policy | `attributes` sensitive keys masked before logs or evidence exports | Retained while user remains reachable or until marketing profile retention cutoff | Delete with the CDP user erasure workflow when profile is linked to the same user id |
| `customer_channel` | `address`, `metadata`, `enabled`, `verified` | PII / confidential | Marketing reachability | phone/email/open_id/channel addresses masked in responses and logs | Retained while the channel is enabled or needed for consent evidence | Delete or disable channel records during user erasure unless legal evidence requires minimal tombstone |
| `marketing_consent` | `user_id`, `channel`, `consent_status`, `source` | compliance evidence | Marketing policy | `user_id` masked in logs; status/source are internal | Keep consent evidence for at least audit retention window; legal hold overrides deletion | `DataDeletionService` deletes active rows for user erasure after evidence/tombstone capture |
| `marketing_suppression` | `user_id`, `channel`, `reason`, `expires_at` | compliance evidence / confidential | Marketing policy | `user_id` masked; reason does not include raw PII | Keep active suppression until expiry plus audit retention window | `DataDeletionService` deletes rows for approved erasure unless legal hold requires minimal evidence |
| `message_send_record` | `user_id`, `request_payload`, `external_message_id`, `error_message` | PII / compliance evidence | Delivery runtime | Payload, channel ids, token-like fields, phone/email/open_id masked by `PiiMaskingService` | Retained for delivery dispute and incident window, then compacted or deleted | `DataDeletionService` deletes rows by tenant and user id after dry-run |
| `canvas_execution_trace` | `input_data`, `output_data`, `error_msg` | internal / possible PII | Runtime platform | Trace JSON must use `PiiMaskingService` / `DataMaskingUtil` before write or export | 30 days online per retention policy; incident windows may be archived | `DataDeletionService` deletes tenant-scoped trace rows whose input/output/error payloads contain the requested user id |
| `canvas_audit_log` | `operator`, `operator_role`, `action`, `detail`, `ip` | compliance evidence / internal | Compliance audit | `AuditEventService` masks metadata before writing `detail` | Minimum 365 days; legal hold overrides deletion | Not deleted by user erasure except legally approved correction; retain minimal audit evidence |
| `data_source_config` | `url`, `username`, `password`, `driver_class_name`, `created_by` | credential / confidential | Data-source management | `password`, token, secret, and apiKey values masked as `****3456`; stored password encrypted by `SecretCipher` | Retain while connector is active; disabled connectors reviewed during credential rotation | Delete data source config only through privileged admin flow; audit every data-source credential change |
| JWT and app secrets | `CANVAS_JWT_SECRET`, trigger HMAC credentials, Redis/RocketMQ credentials | credential | Platform security | Never return raw values; logs use token/secret masking | Rotate per security policy; no application-table retention | Not part of user erasure; revoke/rotate through incident response |

## Implementation References

- `PiiMaskingService` centralizes phone, email, open_id, token, password, credential, and secret masking.
- `GlobalExceptionHandler` masks client-visible error messages before building `R` responses.
- `AuditEventService` writes masked audit metadata into `canvas_audit_log`.
- `DataDeletionService` supports dry-run and execution for `cdp_user_profile`, `cdp_user_identity`, `cdp_user_tag`, `marketing_consent`, `marketing_suppression`, `message_send_record`, and matching `canvas_execution_trace` rows.
