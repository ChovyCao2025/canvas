# 2000+ Execution 并发目标 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a phased delivery path that first proves and stabilizes 500~1000 execution concurrency under the current architecture, then evolves the platform to support 2000+ stable execution concurrency with explicit execution-lane isolation, state layering, Redis layering, MQ flow separation, and measurable acceptance criteria.

**Architecture:** Keep the current WebFlux + RocketMQ + Disruptor + DAG execution model as the baseline. Phase 1 and Phase 2 focus on capacity proof, observability, infrastructure readiness, and hot-path governance. Phase 3 and Phase 4 introduce architectural evolution only where the current design stops scaling efficiently: execution-lane separation, state persistence layering, Redis role separation, MQ topic separation, and downstream isolation.

**Tech Stack:** Java 21, Spring Boot WebFlux, MyBatis-Plus, MySQL, Redis (Lettuce / StringRedisTemplate), RocketMQ, Disruptor, Micrometer / Prometheus, Groovy, existing Canvas execution engine.

public class ExecutionLaneResolver {

    public ExecutionLane resolve(String triggerType,
                                 String triggerNodeType,
                                 boolean heavyTask,
                                 boolean lightTask) {
        if (heavyTask) {
            return ExecutionLane.HEAVY;
        }
        if (lightTask && "DIRECT_CALL".equals(triggerType)) {
            return ExecutionLane.LIGHT;
        }
        return ExecutionLane.STANDARD;
