# 3000 Execution 并发目标 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

> 文件名保留 `2000-concurrency` 是为了沿用既有规划链路；本文档的正式目标已升级为 **3000 stable execution concurrency**。

**Goal:** Upgrade Canvas execution from a single global concurrency model to a production-ready 3000 execution concurrency model with global budget, lane budget, Redis ZSET atomic admission, retry isolation, heavy-task isolation, queue separation, and measurable rollout gates.

**Architecture:** Keep the current Spring Boot WebFlux + RocketMQ + Disruptor + DAG execution model. Extend the existing `InFlightExecutionRegistry` instead of replacing it: it already uses Redis ZSET for canvas/global in-flight slots and should become the lane-aware admission authority. Add `LIGHT`, `STANDARD`, `HEAVY`, and `RETRY` lanes, isolate retry/heavy traffic, and make 1000/2000/3000 capacity gates explicit.

**Tech Stack:** Java 21, Spring Boot WebFlux, MyBatis-Plus, MySQL, Redis (Lettuce / `StringRedisTemplate`), RocketMQ, Disruptor, Micrometer / Prometheus, Groovy, existing Canvas execution engine.

```md
# 3000 Concurrency Capacity Baseline

## Purpose
Prove Canvas execution capacity in controlled stages before production rollout.

3000 concurrency means cluster-level active Canvas executions. It is not HTTP connections, MQ backlog, DAU, or single-instance concurrency.

- Light DAG
- Standard DAG
- Heavy DAG
- Retry surge
- Slow downstream
- Mixed traffic: LIGHT + STANDARD + HEAVY + RETRY
- Multi-instance deployment

## Gates
- 1000 stable execution concurrency: baseline gate
- 2000 stable execution concurrency: platform readiness gate
- 3000 stable execution concurrency: production target gate

## Metrics to capture
- global active executions
- lane active executions
- admission rejections by lane and reason
- overflow count
- retry count and retry backlog
- request backlog
- Disruptor overflow count
- RocketMQ topic backlog
- MySQL active connections and slow SQL
- Redis p95/p99 latency and hot keys
- downstream timeout rate

## Pass criteria
- stable throughput during the full observation window
- no unbounded backlog growth
- no sustained MySQL / Redis / RocketMQ saturation
- retry and overflow remain bounded
- HEAVY and RETRY pressure does not degrade LIGHT and STANDARD beyond target thresholds
```
