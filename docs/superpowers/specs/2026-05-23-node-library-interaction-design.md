# 节点库展示与插入交互设计

## 背景

当前节点面板实现位于 `frontend/src/components/node-panel/index.tsx`，形态为：

- 左侧折叠面板按分类分组
- 每个节点只展示名称
- 说明通过 hover tooltip 展示
- 主要操作是拖拽到画布

这套交互对熟手可用，但对产品/运营用户存在三个问题：

1. 找节点慢：分类折叠导致先记忆分类、再找节点
2. 看不懂：纯名称不足以区分相近节点，说明藏在 hover tooltip 中
3. 插入成本高：拖入后还要确认位置并处理连线，心智负担偏高

本次设计目标是同时解决以上三点，并保证节点数量继续增长时仍可扩展。

## 用户与目标

第一优先用户：产品 / 运营。

他们的典型特征：

- 不一定熟悉所有节点名称
- 更依赖场景说明而非技术术语
- 希望低成本完成常见搭建操作
- 对“插入到了哪里”比“功能很多”更敏感

本次设计目标：

- 默认先看到全部节点，而不是先理解面板结构
- 节点用途在第一眼可被扫读
- 节点插入位置明确，尽量减少手动连线
- 为熟手保留拖拽效率

## 方案比较

### 方案 A：强化现有左侧折叠列表

做法：

- 保留按分类折叠
- 节点项补充一句短说明
- 增加问号详情

优点：

- 改动最小
- 对现有布局影响小

缺点：

- 仍然要求用户先理解分类结构
- 节点增加后，折叠列表会继续变长
- 无法自然承载“常用节点”和更清晰的插入语义

### 方案 B：默认全量展示 + 搜索/筛选 + 画布内插入

做法：

- 节点库默认展示全部节点
- 顶部用搜索和分类 chips 做过滤
- 常用节点置顶
- 节点库负责“选什么”，画布负责“插到哪里”
- 主插入路径改为连线 `+` 和分支占位点

优点：

- 对产品/运营最直观
- 节点查找和节点落点解耦，认知成本更低
- 能兼容未来节点数量增加

缺点：

- 需要同步重做节点库 UI 和画布插入入口
- 比纯侧栏列表多一层交互设计

### 方案 C：完整节点库弹层 / 抽屉

做法：

- 弱化左侧栏
- 用全屏或大抽屉承载节点浏览、说明、筛选

优点：

- 展示空间最大
- 容易做强说明和 richer preview

缺点：

- 打断画布编辑连续性
- 对日常高频操作偏重
- 对当前产品的工作台式布局不够贴合

## 结论

采用 **方案 B**。

原因：

- 最符合产品/运营“先看全、再过滤、再落位”的使用路径
- 插入位置可由画布上下文决定，不再让用户猜测“会插到哪里”
- 同时保留拖拽，不牺牲熟手效率

## 信息架构

节点选择区域调整为三层：

1. 顶部工具区
2. 常用节点区
3. 全部节点区

### 1. 顶部工具区

包含：

- 搜索框：按节点名称、短说明匹配
- 分类 chips：`全部`、各业务分类

规则：

- 默认选中 `全部`
- 点击某个分类后，只展示该分类节点
- 搜索与分类可叠加

### 2. 常用节点区

目的：

- 缩短高频节点访问路径
- 不要求用户每次进入都重扫全部列表

展示规则：

- 始终位于全部节点区上方
- 采用更轻的 tag / pill 形态，而不是重复完整卡片
- 与当前分类过滤联动；若分类不命中，可隐藏不相关常用节点

数据来源：

- 第一阶段可先基于固定配置或最近高频节点
- 后续可扩展为用户级最近使用 / 最常使用

### 3. 全部节点区

展示原则：

- 默认全量展示，不使用折叠分组
- 每个节点为紧凑条目，不做大卡片
- 一屏内尽量展示更多候选，同时保留可读性

## 节点条目设计

每个节点条目包含：

- 分类色条
- 节点名称
- 一句短说明
- 问号帮助入口

### 视觉规则

- 左侧颜色条必须与节点所属分类颜色一致
- 问号使用小圆圈包裹，作为统一帮助入口
- 节点条目默认白底、细边框，hover 和选中态明显区分
- 条目整体风格保持工具面板质感，贴合现有 Ant Design 工作台

### 文案规则

- 名称：保留节点正式名称
- 短说明：只写用途，不写长句教学
- 问号详情：承载更完整说明、适用场景、注意事项

### 说明层级

说明分成两层：

1. 条目上常驻的一句短说明
2. 点击问号后才显示的详细说明

不采用“完全隐藏说明，只点问号才知道是什么”的方案，因为这会提高首轮识别成本。

## 插入交互

核心原则：**节点库只负责选择节点，画布负责确定插入位置。**

### 主路径 1：连线中点插入

在普通连线中点提供 `+` 入口。

交互流程：

1. 用户点击连线中点 `+`
2. 打开节点选择器（可复用左侧节点库能力的轻量版本）
3. 选择节点后自动改线为 `A -> 新节点 -> B`

适用场景：

- 已有线性流程中插入新步骤
- 产品/运营最容易理解的“在线路中加一个节点”

### 主路径 2：分支占位点插入

对多出口节点，在各分支落点提供可见占位点。

交互流程：

1. 用户拖拽节点到某个分支占位点
2. 松手后自动把该节点接到对应分支

规则：

- 占位点必须与具体分支一一对应
- 占位点文案或位置要让人看出属于哪条分支

### 补充路径：拖入空白区

从节点库拖到空白画布时：

- 仅创建节点
- 不自动连线

用途：

- 新建独立片段
- 供熟手进行自由布局

### 弱化路径：点击插入到当前节点后

该能力不作为主入口展示，但可在单出口节点场景保留。

规则：

- 仅当当前上下文明确时可用
- 不作为产品/运营默认学习路径

原因：

- 用户很难仅通过“点击插入”判断插入落点
- 与“位置先确定、节点后选择”的主认知路径冲突

## 分支节点兼容规则

不能对分支节点无脑执行“插到后面”。

规则如下：

- 单出口节点：可直接在后方或连线上插入
- 多出口节点：必须基于具体分支
- 终止节点：不展示“插到后面”
- 空白画布：只允许空白创建

对分支节点，优先支持：

- 从具体分支占位点拖入
- 从某条具体分支连线的中点 `+` 插入

不采用“默认主分支插入”的方案，避免误操作。

## 组件边界

### NodePanel

职责调整为：

- 展示搜索、分类、常用节点、全部节点
- 提供拖拽源能力
- 提供问号详情入口

不负责：

- 决定插入落点
- 直接修改画布边关系

### Canvas Editor

新增职责：

- 在线路中渲染 `+` 插入入口
- 渲染分支占位点
- 接收来自节点库的拖拽，并根据目标区域自动连线
- 在空白区拖入时仅创建节点

### HoverEdge / Placeholder

建议复用现有 `HoverEdge` 与 `BranchPlaceholderNode` 机制扩展：

- 普通 edge 上新增显式插入入口
- 分支 placeholder 强化可见性和可投放语义

这样可以沿用现有画布编辑器的数据结构，不新造一套插入体系。

## 状态与数据

本次设计以现有 `metaApi.getNodeTypes()` 返回的节点定义为基础，不要求后端新增协议。

前端需要新增的本地状态：

- 搜索关键字
- 当前选中分类
- 常用节点集合
- 当前插入上下文（如：edge / branch placeholder / empty canvas）

常用节点第一阶段可使用本地策略，不强依赖后端接口。

## 错误处理与兜底

### 无明确插入落点

若用户没有命中连线或占位点，只拖到空白区：

- 创建节点
- 不自动连线

### 分支节点误投放

若分支占位点不匹配当前拖拽位置：

- 不执行自动接线
- 保持为普通空白落点创建

### 节点说明缺失

若节点无 description：

- 条目短说明回退为类型名或空状态文案
- 问号详情不应出现异常空白布局

## 测试与验证

### 交互验证

1. 默认进入节点库时，不展开折叠组，直接看到全部节点
2. 点击分类 chips 后，全部节点区只保留匹配分类
3. 搜索名称或说明关键词时，结果正确过滤
4. 问号点击后显示详细说明，不影响拖拽主操作
5. 拖入普通空白区时，仅创建节点，不自动连线
6. 点击普通连线的 `+` 后，选中节点可自动插入并改线
7. 拖入分支占位点时，自动挂到对应分支

### 回归验证

1. 现有直接拖拽创建节点能力仍可用
2. 现有 branch placeholder 逻辑不被破坏
3. 现有 edge 渲染与删除逻辑不被破坏

## 不在本次范围

- 节点说明体系的内容治理
- 用户级“最近使用 / 最常使用”后端统计
- 节点库大弹层 / 全屏资源中心
- 节点图标系统重构
- 多语言文案体系

## 实施建议

按以下顺序推进：

1. 节点库展示改造：默认全量、分类 chips、常用节点、条目样式
2. 问号详情与短说明层级
3. 普通连线 `+` 插入
4. 分支占位点自动连线规则收敛
5. 交互细节打磨与回归测试
