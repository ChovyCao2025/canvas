# 2000+ Execution 并发目标 Design Spec

## 1. 背景

当前 Canvas 系统已经具备继续从 500 向 1000 execution 并发上探的基础能力，但 500~1000 区间的主要矛盾，仍然集中在：
- MySQL / Redis / RocketMQ / 下游服务的配套资源是否足够
- 压测、监控、调优是否到位
- 热点治理与重任务治理是否完成

因此，500~1000 更准确地说是“运维配套与治理问题”，不能简单定义为“代码能力不够”。

但当目标进一步上探到 **2000+ 稳定 execution 并发** 时，单纯依赖扩应用、扩基础设施和一般性调优，收益会开始明显下降。此时需要把问题从“容量扩容”上升到“平台级架构演进”。

本设计采用**分层演进型方案**：
- 500~1000：以运维配套、热点治理、热路径治理为主
- 2000+：在现有架构基础上，演进为更明确的分层执行平台

---

## 2. 目标定义

### 2.1 总目标
建设一套支持 **2000+ 稳定 execution 并发** 的平台能力。

### 2.2 目标口径
- execution 并发 = 高峰时刻同时执行的画布执行数
- 不是 HTTP 连接数
- 不是总客户数
- 不是简单的 MQ 堆积能力

### 2.3 对应用户规模口径
2000 execution 并发大致对应：
- **50万 ~ 100万+ DAU（中性口径）**
- **50+ 中型客户规模（粗估）**

其中：
- 如果客户峰值更集中，则实际可承载客户数会下降
- 如果流量更分散、链路更轻，则可承载用户规模会更高

### 2.4 成功标准
1. 在约定链路模型下达到 2000+ 稳定 execution 并发
2. 主链路 RT、错误率、积压在可接受阈值内
3. 轻任务不会被重任务拖垮
4. 容量提升不是单纯依赖基础设施堆料，而是具备明确的平台扩展路径

---

## 3. 当前边界

## 3.1 当前系统仍具备继续上探的基础
当前代码之所以还可以继续从 500 向 1000 上探，原因在于：
- WebFlux 入口层具备较好的连接承接能力
- MQ + Disruptor 已具备一定削峰与异步分发能力
- 执行引擎已有 execution 并发准入、overflow/retry 等基础控制能力
- 虚拟线程对部分阻塞型 Handler 有缓冲作用

因此，当前系统不是“只能做到 500”。

## 3.2 当前系统的主要代码边界
代码层边界主要体现在：
- 主链路中 Redis / MySQL 往返次数较多
- execution / request / quota 状态写入频繁
- 轻任务与重任务还没有完全通道隔离
- Redis 同时承担 ctx / dedup / lock / quota / route / rate limit 等职责
- retry / replay / audience / groovy 仍共享较多资源
- 下游依赖虽有连接池，但隔离与治理粒度不够细

这些问题在 500~1000 区间未必都会成为决定性障碍，但在 2000+ 时会放大。

## 3.3 当前系统的主要运维边界
运维与配套实施边界主要体现在：
- MySQL 规格是否足够
- Redis 是否高配或分层
- RocketMQ broker 是否足够
- 下游服务是否一起扩容
- 应用实例数是否合理
- 压测与监控是否完善

这些不是代码缺陷，但决定了代码能力能不能被真正发挥出来。

## 3.4 为什么 2000+ 不能继续只靠运维放大
到 2000+ 并发时，系统主要矛盾会从“资源不够”转成“架构边界不够清晰”：
- 在线执行状态写入仍然过重
- Redis 职责过多，互相干扰
- 轻重任务混跑，导致容量不可预测
- retry / replay / audience / groovy 更容易冲击主链路
- 下游长尾会更明显地传染到主链路

结论：
- 500~1000：更多是运维配套与治理问题
- 2000+：需要平台级架构演进

---

## 4. 方案比较

## 4.1 方案 A：分层演进型（推荐）
- 500~1000：以运维配套、热点治理、热路径治理为主
- 2000+：在现有架构基础上做执行通道拆分、状态分层、Redis 分层、MQ 流量分治
- 保留现有 WebFlux + MQ + Disruptor + DAG 执行主结构

优点：
- 对现有系统改动可控
- 适合阶段推进
- 能明确区分 500~1000 与 2000+ 的问题边界

缺点：
- 系统将从单一执行模型演进成多通道平台模型，复杂度会提升

## 4.2 方案 B：纯运维放大型
- 尽量不改代码架构
- 依赖更多应用实例、更大 MySQL/Redis/RocketMQ、更强下游能力

优点：
- 短期动作最少

缺点：
- 到 2000+ 收益明显递减
- 热点与重任务问题不会根治
- 容量会越来越依赖基础设施堆料

## 4.3 方案 C：大改重构型
- 直接把执行平台拆成多服务平台
- 实时执行、重型任务、状态服务、调度服务彻底独立

优点：
- 长期上限最高

缺点：
- 当前阶段成本最高
- 对现有系统侵入最大
- 不适合作为第一步

## 4.4 推荐结论
推荐采用 **方案 A：分层演进型**。

原因：
1. 与当前系统现状最匹配
2. 更符合“500~1000 主要是运维配套问题，2000+ 才是架构问题”的判断
3. 能保留当前系统已有能力，避免过早重构

---

## 5. 架构演进目标

## 5.1 执行通道拆分
将执行层拆成至少 3 类通道：
1. **轻量实时通道**
   - 短 DAG
   - 少量 Redis / DB
   - 少量下游调用
   - 目标：低延迟
2. **标准异步通道**
   - 正常营销画布
   - 中等复杂度 DAG
3. **重型任务通道**
   - audience 计算
   - groovy 重逻辑
   - 大规模 replay
   - 大批量外部调用

目标：
- 轻任务不再被重任务拖慢
- 集群可按任务类型独立扩容

## 5.2 状态模型分层
把状态拆成：
1. **强在线状态**
   - 执行必须依赖
2. **弱在线状态**
   - 控制台秒级可见即可
3. **分析/审计状态**
   - 不再与在线执行主表竞争

目标：
- 减轻 MySQL 热表压力
- 降低同步写放大

## 5.3 Redis 分层
至少拆为：
1. **执行态 Redis**
   - ctx / dedup / lock / quota / replay
2. **缓存/路由 Redis**
   - route / cache / pubsub
3. **bitmap / 大对象 Redis**
   - audience bitmap / 大中间结果

目标：
- 避免大对象拖慢在线小 key
- 避免 route/cache 干扰主链路

## 5.4 MQ 流量分治
按用途拆分：
- 正常流量 topic
- retry / overflow topic
- 重型任务 topic

目标：
- 高峰期优先保主链路
- 不让 retry 洪峰冲击正常流量

## 5.5 下游依赖平台化治理
- 按依赖类型独立连接池
- 熔断 / 限流 / 超时治理
- 容量契约化

目标：
- 把下游长尾限制在局部
- 提升系统整体可预测性

---

## 6. 分阶段实施路径

## Phase 1：把 1000 做稳
目标：
- 通过压测和调优，把 500~1000 区间边界做清楚并稳定住

重点：
- 压测
- 监控
- 运维配套提升
- 热路径治理

## Phase 2：完成 2000 的架构准备
目标：
- 完成执行通道拆分设计
- 完成状态分层设计
- 完成 Redis 分层设计
- 完成 MQ 分治设计

重点：
- 技术方案
- 边界设计
- 演进路径设计

## Phase 3：交付 2000+ 平台能力
目标：
- 基于新架构完成实现、验证和上线

重点：
- 分阶段实施
- 平滑迁移
- 上线与回滚可控

---

## 7. 验收标准

## 7.1 性能指标
- 稳定 execution 并发达到 2000+
- 核心链路 RT 可控
- 错误率低于约定阈值
- retry / overflow / backlog 在可控范围

## 7.2 基础设施指标
- MySQL 连接、慢 SQL、热点表压力可控
- Redis RT、热点 key、连接数可控
- RocketMQ 无持续积压
- 下游依赖长尾受控

## 7.3 架构指标
- 轻任务与重任务已隔离
- 状态分层完成
- Redis 分层完成
- MQ 正常流量与 retry 流量隔离完成

---

## 8. 风险与非目标

### 8.1 风险
- 平台复杂度上升
- 集群与依赖运维复杂度上升
- 分阶段演进期间需控制兼容与迁移风险

### 8.2 非目标
本设计不要求：
- 立即推翻现有系统重建
- 一步到位做成完全独立多服务平台
- 在没有容量基线和压测结果前直接承诺 2000+
