# Product Platform Suite Design

## 背景

当前项目已经具备营销旅程画布、执行引擎、灰度/回滚、dry-run、CDP 用户中心、标签、人群、触发源配置、执行统计和通知能力。系统已经从单一画布工具演进为营销自动化平台。

本轮目标不是继续补零散页面，而是把项目收敛为一个可继续 SaaS 化的产品平台：

- 上层：运营旅程配置、上线、观察和异常处理闭环。
- 中层：执行稳定性、积压治理、重放保护和容量风险管理。
- 底层：租户、角色、配置隔离和用量统计基础。

CDP 深水区能力暂不作为本轮实施重点，现有 CDP 用户中心和标签闭环保持兼容。

## 总体定位

项目定位为：CDP 驱动的营销自动化 SaaS 平台。

本轮先实施三条产品路线：

1. 运营闭环。
2. 平台稳定性。
3. SaaS 基座。

三条路线使用独立 git worktree 开发，最终通过集成分支合并。

## Worktree 边界

### `feat/saas-foundation`

负责共享底座：

- 租户模型。
- 用户角色扩展。
- JWT 租户与角色声明。
- 核心业务表 `tenant_id` 迁移。
- 租户级配置覆盖。
- 租户用量统计基础。

这是另外两条分支的前置依赖。

### `feat/platform-stability`

负责执行平台治理：

- 执行请求控制台。
- 积压、溢出、retry、DLQ、MQ rejected 可视化。
- 限流与熔断配置入口。
- 容量风险指标。
- replay、overflow、批量任务的主链路保护。

该分支不拆执行引擎为多个服务。

### `feat/ops-closed-loop`

负责运营产品闭环：

- 运营中心。
- 发布审批入口。
- 异常与 DLQ 处理控制台。
- 模板中心前端接入。
- 高风险动作审计可见性。

该分支不做复杂 BPMN 审批流。

### `feat/product-platform-suite`

最终集成分支：

- 合并三条 worktree 的成果。
- 统一前端导航、权限菜单、共享 API 类型、数据库迁移顺序。
- 跑完整构建和关键链路验证。

合并顺序：

1. `feat/saas-foundation`
2. `feat/platform-stability`
3. `feat/ops-closed-loop`

## 共享数据模型

### 租户

新增 `tenant` 主表：

- `id`
- `name`
- `status`
- `plan_code`
- `quota_json`
- `remark`
- `created_by`
- `created_at`
- `updated_by`
- `updated_at`

租户状态固定为：

- `ACTIVE`
- `DISABLED`

### 用户与角色

扩展现有 `sys_user`：

- 增加 `tenant_id`
- 角色扩展为 `SUPER_ADMIN`、`TENANT_ADMIN`、`OPERATOR`

登录 JWT 增加：

- `tenantId`
- `role`
- `username`
- `userId`

第一阶段只支持一个用户归属一个租户，不做多租户切换。

### 业务表租户隔离

核心业务表增加 `tenant_id`：

- `canvas`
- `canvas_version`
- `canvas_execution`
- `canvas_execution_trace`
- `canvas_execution_request`
- `canvas_execution_dlq`
- `canvas_mq_trigger_rejected`
- `api_definition`
- `event_definition`
- `mq_message_definition`
- `tag_definition`
- `tag_value_definition`
- `audience_definition`
- `audience_compute_run`
- `cdp_user_profile`
- `cdp_user_identity`
- `cdp_user_tag`
- `cdp_user_tag_history`
- `cdp_tag_operation`
- `notification`
- `async_task`
- `system_option`
- `canvas_template`

迁移现有数据时统一归属 `default` 租户。

### 审计

高风险动作必须写审计记录：

- 发布。
- 灰度。
- 晋升灰度。
- 回滚。
- Kill。
- 审批通过/拒绝。
- DLQ 重放。
- 批量重放。
- 限流/熔断配置变更。
- 租户配额变更。

审计字段至少包含：

- `tenant_id`
- `operator`
- `action`
- `target_type`
- `target_id`
- `before_json`
- `after_json`
- `reason`
- `created_at`

## 权限模型

### `SUPER_ADMIN`

平台管理员：

- 管理租户。
- 查看所有租户用量。
- 按租户筛选查询数据。
- 调整租户配额。
- 管理官方模板。

### `TENANT_ADMIN`

租户管理员：

- 管理本租户用户。
- 管理本租户配置。
- 发布、回滚、灰度、Kill 本租户画布。
- 审批本租户高风险发布。
- 执行本租户批量重放。

### `OPERATOR`

运营人员：

- 编辑本租户旅程。
- 查看本租户数据。
- 提交高风险发布审批。
- 查看异常和执行结果。

默认不能执行高风险动作。

### 隔离策略

- 后端从 JWT 解析 `tenantId` 和角色。
- 所有列表和详情查询默认追加 `tenant_id` 条件。
- 写入接口由服务层从登录态注入 `tenant_id`，前端传入值不可信。
- 只有 `SUPER_ADMIN` 可以显式按租户筛选。
- 租户内模板可以复制和编辑，官方模板由 `SUPER_ADMIN` 管理。

## 运营闭环设计

### 运营中心

新增统一入口，展示：

- 待审批发布。
- 异常执行。
- DLQ。
- MQ 触发拒收。
- 重放任务。
- 最近告警。

首页保持概览，运营中心负责处理动作。

### 发布审批

高风险画布发布进入审批流。高风险规则第一阶段使用可解释规则：

- 含发券节点。
- 无全局总量上限。
- 含 Groovy 节点。
- 含触达类节点。
- 灰度比例大于 50%。

审批列表支持：

- 查看画布、版本、风险原因。
- 通过。
- 拒绝。
- 填写审批意见。

非高风险发布仍允许直接发布。

### 异常与 DLQ 控制台

展示：

- DLQ。
- 执行失败。
- MQ 触发拒收。
- 执行请求积压。

支持筛选：

- 画布。
- 用户。
- 错误类型。
- 时间范围。
- 状态。

支持动作：

- 单条重放。
- 条件批量重放。

批量重放需要 `TENANT_ADMIN` 权限。

### 模板中心

接入已有接口：

- `GET /canvas/templates`
- `POST /canvas/{id}/save-as-template`
- `POST /canvas/from-template/{templateId}`

支持：

- 从模板新建旅程。
- 当前旅程另存为模板。
- 官方模板只读。
- 租户模板可编辑或停用。

### 审计可见性

在画布详情或运营中心展示高风险动作审计：

- 谁操作。
- 操作时间。
- 动作类型。
- 目标对象。
- 操作前后摘要。
- 原因。

本轮不建设完整日志平台。

## 平台稳定性设计

### 执行请求控制台

接入已有 `/canvas/execution-requests` 能力，展示：

- 请求状态。
- 来源。
- 优先级。
- 重试次数。
- 错误原因。
- 创建时间。
- 更新时间。

支持筛选：

- 租户。
- 画布。
- 状态。
- 来源。
- 时间范围。

支持单条重放和条件批量重放。

### 积压与溢出视图

展示数量和趋势：

- 正常执行请求。
- retry 请求。
- overflow 请求。
- 手工 replay 请求。
- MQ rejected。
- DLQ。

首页和运营中心展示摘要，稳定性页面展示明细。

### 限流与熔断配置入口

优先扩展现有 `system_option` 为租户级治理配置。配置按 `tenant_id + category + option_key` 唯一；`tenant_id` 为空表示平台默认值。

第一阶段提供：

- 画布级 QPS 上限。
- 画布级并发上限。
- 节点类型级超时。
- 节点类型级重试上限。
- 外部依赖级熔断开关。
- 外部依赖级失败阈值。
- 外部依赖级恢复窗口。

配置变更写审计日志。

第一阶段保存后生效，不做实时灰度配置中心。

### 容量风险指标

展示：

- 执行中数量。
- 失败率。
- 平均耗时。
- 长尾耗时。
- 重试率。
- 积压量。
- DLQ 增长量。

风险分级：

- 正常。
- 观察。
- 风险。
- 阻断。

规则使用简单可解释阈值，不做预测模型。

### 主链路保护

对以下任务打上任务类型：

- replay。
- overflow。
- audience 计算。
- Groovy 重逻辑。
- 批量外部调用。

前端默认限制单次批量重放最多 100 条；若租户配额配置了更低上限，使用租户上限。

后端执行同样的规模校验，并对 replay 和 overflow 使用独立限流配置，避免冲击正常触发。

## SaaS 基座设计

### 租户管理

`SUPER_ADMIN` 可：

- 创建租户。
- 停用租户。
- 恢复租户。
- 调整套餐标识。
- 调整配额 JSON。
- 查看租户用量。

现有数据迁移到 `default` 租户。

### 用户管理

`SUPER_ADMIN` 可管理所有租户用户。

`TENANT_ADMIN` 只能管理本租户用户。

`OPERATOR` 无用户管理权限。

### 租户级配置

系统选项支持租户级覆盖：

- 存在租户配置时使用租户配置。
- 不存在租户配置时使用默认配置。

API、MQ、事件、标签、人群配置归属租户。

官方示例模板可复制到租户内，复制后成为租户资产。

### 用量统计基础

按租户统计：

- 画布数。
- 发布画布数。
- 执行次数。
- 失败次数。
- 用户数。
- 重放次数。
- DLQ 数量。

第一阶段不做账单，只提供 SaaS 后续计费和配额判断的数据基础。

配额超限时先提示并阻断高风险动作：

- 新建画布。
- 发布。
- 批量重放。

## 非目标

本轮不做：

- 完整 CDP 画像和事件平台。
- 多组织层级。
- 复杂 RBAC 策略引擎。
- SSO。
- 支付、发票和套餐购买。
- 跨租户数据共享。
- 多级 BPMN 审批。
- 完整 APM 平台。
- 执行引擎微服务拆分。
- Redis/MQ 物理分层自动化。
- 形式主义单测和快照测试。

## 测试策略

测试只覆盖风险点。

### 必测

- 默认租户迁移后现有数据可用。
- 普通租户不能读取其他租户画布。
- 普通租户不能读取其他租户执行记录、DLQ、配置和通知。
- `SUPER_ADMIN` 能按租户筛选。
- `TENANT_ADMIN` 不能管理其他租户用户。
- 高风险发布不能绕过审批。
- 审批通过后才能发布对应版本。
- 审批拒绝后不能发布对应版本。
- DLQ 单条重放写审计。
- 批量重放受权限和规模限制。
- 限流/熔断配置变更写审计。
- replay/overflow 限流不影响正常触发配置。
- 模板创建出的旅程可编辑并归属当前租户。

### 少测或不测

- DTO getter/setter。
- 无分支 CRUD 包装。
- 纯展示组件快照。
- 只验证文案的 UI 单测。
- 无业务分支的 Mapper 调用。

前端以关键页面 smoke 和关键纯函数测试为主。后端以服务层和 Controller 边界测试为主。

## 验收标准

### 运营闭环

- 运营能看到所有待处理风险项。
- 高风险发布不会绕过审批。
- 异常执行能被定位、筛选、重放。
- 模板能创建可编辑旅程。
- 审计能回答谁在什么时候做了什么高风险动作。

### 平台稳定性

- 平台管理员能看到当前是否积压、哪里失败、是否需要限流。
- 重放和批量任务不能轻易拖垮正常执行。
- 关键治理配置可以在 UI 管理并审计。
- 风险指标能指导扩容或止血动作。

### SaaS 基座

- 现有数据迁移后仍能登录和使用。
- 普通租户无法读写其他租户数据。
- 平台管理员可以管理租户和查看租户用量。
- 租户级配置不污染全局默认配置。
- 运营闭环和平台稳定性分支可以基于 `tenant_id` 接入。

## 实施顺序

1. 创建 `feat/saas-foundation` worktree，完成共享底座。
2. 基于共享底座创建或同步 `feat/platform-stability` worktree。
3. 基于共享底座创建或同步 `feat/ops-closed-loop` worktree。
4. 创建 `feat/product-platform-suite` 集成分支。
5. 依次合并三条分支。
6. 处理迁移顺序、前端导航、权限菜单和共享类型冲突。
7. 跑完整构建和关键链路验证。

## 风险与处理

### 现有工作区很脏

使用独立 worktree 开发，不回滚当前工作区已有改动。

### 租户字段影响范围大

先交付 SaaS 基座，再让其他分支接入统一模型。

### 数据迁移风险

所有历史数据归入 `default` 租户，先保证兼容，再逐步开放租户管理。

### 测试成本失控

只测隔离、权限、状态流转、重放保护和高风险副作用，不追求形式覆盖率。

### 三条分支冲突

共享模型只在 `saas-foundation` 定义。其他分支不重复新增租户字段和角色枚举。
