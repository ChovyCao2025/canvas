# 节点配置侧边栏视觉升级设计

## 背景

当前画布编辑页右侧的节点配置侧边栏已经具备完整功能，但整体仍接近默认 Ant Design 表单直出，主要问题是：

- 信息层级过平，节点类型、节点标题、核心配置、分支结果都落在同一视觉平面
- 核心配置项缺少摘要表达，第一眼更像“字段列表”而不是“节点配置面板”
- `TAGGER` 这类有明显业务语义的节点没有被识别性地表达出来
- “命中分支 / 未命中分支”与普通字段混在一起，流程去向不够直观

这次工作只做右侧节点配置侧边栏的 UI 升级，不改变当前配置能力、字段语义、保存逻辑和后端接口行为。

## 目标

- 将右侧侧边栏从“默认表单堆叠”升级为更成熟的 inspector 风格配置面板
- 强化节点身份识别，让用户先理解“我正在配置哪个节点”
- 强化核心配置和分支结果的层级，让用户更快完成阅读和编辑
- 在 `TAGGER` 节点上做定向视觉增强，提升识别度和产品感
- 确保现有功能行为零回归，只升级 UI 展示

## 非目标

- 不修改节点 schema 来源和字段定义
- 不修改 `onChange`、保存、联动、可见性判断等业务逻辑
- 不增加新的后端接口字段或前端持久化状态
- 不改左侧节点库、不改画布节点卡片本体、不改画布主编辑区布局
- 不引入新的复杂交互流程，如多步表单、tab 化配置、二级弹窗等

## 约束

- 现有功能不能改坏，视觉升级不能影响已存在的配置、保存和节点切换行为
- 当前项目这块以局部内联样式和组件内布局为主，本次实现优先遵循现有模式
- 侧边栏宽度有限，设计必须优先保证桌面端窄栏可读性
- 通用样式必须兼容多种节点类型，不能只对 `TAGGER` 节点成立

## 方案选择

讨论过的方向有三类：

1. 保守增强：保留当前表单形态，只做间距和层级优化
2. 中度重构：增加摘要卡和分支模块，但整体仍偏“表单面板”
3. 明显升级：整体改成 inspector 风格，并对 `TAGGER` 做识别性增强

最终采用 **方案 3 的结构强度 + 方案 1 的实现边界**：

- 视觉上明显升级，不停留在“调间距”
- 实现上不碰业务逻辑，只在 `ConfigPanel` 内部重组展示结构
- 通用节点先获得统一 inspector 样式
- `TAGGER` 节点额外获得强化识别头部和分支展示

## 结构设计

右侧节点配置侧边栏统一改造成三段式结构：

1. 顶部识别区
2. 内容配置区
3. 分支结果区

整体视觉基调：

- 白底为主
- 浅边框卡片
- 统一圆角，建议 `14px - 18px`
- 使用很轻的阴影，不做厚重浮层感
- 蓝色作为主强调色，红色只用于失败 / 未命中 / 异常分支
- 字体层级收紧，避免标题、标签、说明同时抢注意力

不使用高饱和渐变背景、大面积深色头图或偏营销化的装饰。

## 顶部识别区

### 通用节点

所有节点都应先显示一个统一的 header card，用来回答：

- 当前是什么节点类型
- 当前节点名称是什么
- 当前是否已经配置

通用结构为：

- 节点类型小标签
- 节点主标题
- 可选状态标签，如“已配置”
- 弱化的一句说明或节点分类信息

其中节点类型标签和状态标签都是辅助信息，不能压过节点名称本身。

### `TAGGER` 节点强化样式

`TAGGER` 节点使用已经确认的 A 方向收敛稿：

- 左上保留 `Tagger` 胶囊标签
- 主标题使用节点名称，例如 `是否高价值近30天活跃用户`
- 主标题下使用双标签信息，用于概括节点模式和身份
- 说明文案保留，但字号更小、颜色更淡，只做辅助说明

最终头部结构如下：

1. `Tagger` 胶囊标签
2. 节点标题
3. 双标签信息
4. 弱说明文案
5. 右侧轻量状态标签

视觉要求：

- `Tagger` 胶囊要明显，但不能变成整块重色 banner
- 标题字号比普通字段明显更大，但控制在窄栏内不显压迫
- 双标签信息需要帮助快速理解，不承担导航或操作职责
- 说明文案必须弱化，不能与标题竞争层级

## 内容配置区

内容配置区不再表现为连续无结构的 `Form.Item` 列表，而是改成分组卡片 + 摘要字段的形式。

### 通用规则

- 每组配置放入独立卡片或卡片内分段
- 组标题使用 `12px - 13px` 的弱标题样式
- 组内字段保留原有编辑控件，不改变表单行为
- 核心字段优先做“可编辑摘要”式展示，次要字段仍可保持普通表单行

这里的“可编辑摘要”不是只读信息块，而是：

- 先通过标签、标题、边框和卡片感提高识别度
- 控件仍然直接可见、直接可编辑
- 不增加额外的展开、切换、二次点击路径

### `TAGGER` 节点内容优先级

`TAGGER` 节点中优先强化以下字段：

- `标签模式`
- `人群`

这两个字段优先展示为摘要卡：

- 上方是弱标签名
- 下方是更清晰的当前值
- 保留可编辑能力

目的是让用户在进入面板时，不需要从多个下拉框中逐个读字段名，直接先看到“当前模式是什么、选的是谁”。

### 其他节点兼容

对于非 `TAGGER` 节点，不强行套用相同摘要结构，但仍统一得到：

- 更清晰的顶部 header
- 更稳定的卡片边界
- 更统一的字段间距
- 更克制的说明层级

这样可以避免为了一类节点牺牲通用性。

## 分支结果区

### 目的

分支结果区单独成段，是这次改造的重要变化。它的目的是让用户快速理解：

- 命中后流向哪里
- 未命中后流向哪里
- 当前是否存在未连接分支

### 结构

对 `TAGGER` 节点，分支区放在内容配置区之后，作为独立结果块展示。

每个分支使用独立状态卡：

- 左侧是分支名称
- 右侧是目标节点名称
- 使用轻蓝表示命中分支
- 使用轻红表示未命中分支

如果分支未连接，明确显示 `未连接`，不能留空。

### 视觉要求

- 色彩只作为辅助识别，不变成高饱和警示块
- 目标节点名称比分支标签更显眼
- 分支卡之间使用稳定垂直间距，避免贴得太紧
- 文案长度较长时允许换行，但整体卡片不能破坏节奏

## 交互规则

- 不新增 tabs、折叠面板或 modal 级交互
- 顶部识别区始终在面板顶部优先展示
- 表单控件仍保持原有编辑路径，不做“摘要态 / 编辑态”切换
- 节点切换时只更新当前面板内容，不引入新的异步加载路径
- 分支区仅增强展示，不改变节点连线本身的配置方式

## 实现边界

本次改动尽量收敛在 [frontend/src/components/config-panel/index.tsx](/Users/photonpay/project/canvas/frontend/src/components/config-panel/index.tsx)。

建议在该文件内新增或抽出几个轻量展示组件：

- `NodeHeaderCard`
- `ConfigSectionCard`
- `FieldSummaryRow`
- `BranchRouteCard`

这些组件只负责布局和视觉表达，不新增业务状态源。

必须保留的现有机制：

- `schema` 驱动字段渲染
- `evaluateVisible`
- `handleValuesChange`
- `renderControl`
- 当前 `Form` 数据同步逻辑

也就是说，这次是“展示层重组”，不是“配置逻辑重写”。

## 样式策略

- 优先使用组件内样式常量或局部样式对象
- 不新增全局主题覆盖
- 不引入新的样式库
- 保持与当前项目既有内联样式组织方式一致

如果后续侧边栏视觉模式在更多地方复用，再考虑抽离成独立样式层。

## 风险与控制

### 风险 1：视觉重构时误伤表单行为

控制方式：

- 不改字段 `name`
- 不改 `Form.Item` 的校验规则
- 不改 `onValuesChange` 数据流
- 每次只重排容器层级，不改字段语义

### 风险 2：只对 `TAGGER` 好看，其他节点变得拥挤或奇怪

控制方式：

- 先定义通用 inspector 外壳，再对 `TAGGER` 做增强
- 验证至少 2 至 3 个非 `TAGGER` 节点类型
- 避免把过强的头部结构硬套给所有节点

### 风险 3：窄侧栏下出现换行、拥挤或滚动异常

控制方式：

- 标题、标签、说明都做宽度和换行约束
- 分支卡允许长文案自然换行
- 保持单列纵向滚动，不增加横向布局负担

## 验证方案

### 功能零回归

必须确认以下行为与改造前一致：

- 节点切换后正确加载当前配置
- 输入、选择、开关变更后仍正确触发 `onChange`
- 可见性条件字段仍正确显示 / 隐藏
- 只读态 `readonly` 仍正确生效
- 原有复杂控件如 `api-input-params`、`event-attr-preview` 不受破坏

### 视觉验收

至少覆盖以下场景：

1. `TAGGER` 节点，已完整配置
2. `TAGGER` 节点，存在未连接分支
3. 至少 2 至 3 个非 `TAGGER` 节点
4. 长节点名称场景
5. 窄侧栏高度滚动场景

### 验收标准

- 用户第一眼能分辨节点身份、核心配置和分支去向
- `TAGGER` 头部识别明显提升，但整体仍保持后台工具的克制感
- 说明文案被正确弱化，不抢主层级
- 改造后没有新增功能回归或配置错误

## 最终结论

本次节点配置侧边栏优化采用 inspector 风格重组，核心是：

- 用顶部识别区解决“在配什么”
- 用内容配置区解决“当前配了什么”
- 用分支结果区解决“接下来会去哪”

其中 `TAGGER` 节点使用已确认的组合头部样式做重点增强，但整个方案仍以“只升级 UI，不改现有功能行为”为最高约束。
