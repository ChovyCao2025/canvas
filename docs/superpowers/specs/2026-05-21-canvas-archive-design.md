# 画布归档功能设计（优化点 #1）

## 背景

画布列表目前没有删除入口，用户误创建后无法清理。本方案以软删除（归档）方式解决，保留数据、支持管理员恢复，同时不影响正在运行的 execution。

## 范围

- **本版本**：用户归档画布，管理员手动恢复（改库）
- **后续迭代**：用户自助查看并恢复已归档画布

---

## 后端设计

### 枚举扩展

在 `CanvasStatusEnum` 中加入：

```java
ARCHIVED(3);
```

现有值：DRAFT=0, PUBLISHED=1, OFFLINE=2, KILLED=4，值 3 原本空缺，直接填入。

### 新接口

```
POST /canvas/{id}/archive
```

- 权限：创建人或管理员
- 状态机：DRAFT / PUBLISHED / OFFLINE / KILLED → ARCHIVED 均允许
- 执行语义：仅将 `canvas.status` 改为 3，不中断已运行的 execution

### 列表查询调整

`CanvasService.list()` 默认过滤归档项：

```java
.ne(Canvas::getStatus, CanvasStatusEnum.ARCHIVED.getCode())
```

若传入 `status=3` 则精确查询归档列表（为后续自助恢复预留接口兼容性）。

### 错误响应

| 场景 | HTTP 状态 | 说明 |
|------|-----------|------|
| 画布已归档 | 400 | 幂等保护 |
| 非创建人且非管理员 | 403 | 权限拒绝 |

---

## 前端设计

### 入口：列表页操作列

在每行操作列加"更多"下拉菜单（`Dropdown`），菜单项：**归档画布**（红色文字）。

### 确认弹窗（B 风格）

```
⚠ 归档画布
确认将「{canvas.name}」归档？
  · 画布将从列表中隐藏
  · 正在运行中的流程不受影响
  · 可联系管理员恢复
[取消]  [确认归档]
```

### 归档成功后

- 列表移除该行（无需刷新整页）
- Toast 提示："已归档"

### 错误处理

| 场景 | 前端处理 |
|------|---------|
| 400 已归档 | Toast："画布已归档" |
| 403 无权限 | Toast："无操作权限" |
| 网络失败 | 关闭弹窗，Toast："归档失败，请重试"，列表不变 |

---

## 数据流

```
用户点击"更多 → 归档画布"
  → 弹出确认弹窗
  → 点击"确认归档"
  → POST /canvas/{id}/archive
  → 后端 status 改为 ARCHIVED(3)
  → 前端移除该行 + Toast "已归档"
```

---

## 不在本版本范围内

- 已归档画布的列表页（用户自助查看）
- 用户自助恢复接口
- 批量归档
