# Canvas 系统并发能力评估报告

## 1. 结论摘要

### 1.1 先给结论
- 当前系统**不能直接按配置里的 `max-concurrency=1000` 对外承诺**。
- 更合理的口径应该是：
  - **4C / 8G 单实例**：建议按 **80 ~ 150** 稳定 execution 并发规划
  - **8C / 16G 单实例**：建议按 **150 ~ 300** 稳定 execution 并发规划
- **500 并发**：当前代码 + 合理运维配套 + 调优治理，可以实现
- **1000 并发**：当前代码基础上，在较强运维配套和一定代码治理下，有机会实现
- **2000+ 并发**：不能只靠扩机器和扩配套资源，建议进入平台级架构演进

### 1.2 最重要的一句话

> 当前系统的上限，不应只看应用机器规格，也不应只看 YAML 配置，而要看“代码实现能力 + MySQL/Redis/RocketMQ/下游服务配套能力”这个整体。

---

## 2. 怎么看“代码能力”与“运维配套能力”

这次评估需要先分清两类边界。

### 2.1 代码实现能力边界
这部分是代码本身决定的：
- 主链路里 Redis / MySQL 往返次数较多
- execution / request / quota 在线写入较频繁
- 轻任务与重任务还没有完全分通道隔离
- Redis 承担了 ctx / dedup / lock / quota / route / rate limit 等多重职责
- retry / replay / audience / groovy 仍然共享很多执行资源
- 下游 HTTP 依赖虽然有连接池，但隔离粒度还不够细

这些属于**代码/架构层能力边界**。

### 2.2 运维与配套实施边界
这部分不是代码缺陷，而是系统落地条件：
- MySQL 规格够不够
- Redis 是否高配或分层
- RocketMQ broker 是否足够
- 下游接口是否一起扩容
- 应用实例数是否合理
- 监控是否齐全
- 压测是否做过

这些属于**运维与配套实施要求**，不能简单算成“代码不行”。

### 2.3 结论
所以更准确的说法是：

- **500 ~ 1000 区间**：更多是“运维容量 + 配套资源 + 调优治理”问题
- **2000+ 区间**：才更应该上升为“平台级架构演进”问题

## 4. 单实例能力建议

| 应用规格 | 建议稳定 execution 并发 | 激进尝试值 | 说明 |
|---|---:|---:|---|
| 2C / 4G | 30 ~ 80 | 100 | 适合低流量、轻链路 |
| 4C / 8G | 80 ~ 150 | 200 ~ 250 | 当前最现实生产起点 |
| 8C / 16G | 150 ~ 300 | 300 ~ 500 | 需要更强配套资源 |
| 16C / 32G | 250 ~ 500 | 500+ | 更推荐横向扩容 |

说明：
- 这里说的是**稳定 execution 并发**
- 不是 HTTP 连接数，也不是理论峰值
- 如果链路偏重，应按区间下沿甚至更保守值规划

## 6. 并发目标对应的用户规模 / 客户规模

### 6.1 核心原则

> execution 并发不是客户数，而是高峰时刻同时触发执行的用户数。

所以：
- `500 并发` ≠ `500 个客户`
- 它更接近于：系统在高峰时刻同时处理 500 个用户触发的执行请求

### 6.2 估算表

| execution 并发目标 | 高峰同时触发用户数 | 保守场景 DAU | 中性场景 DAU | 理想场景 DAU | 对应中型客户数（粗估） | 说明 |
|---:|---:|---:|---:|---:|---:|---|
| 500 | 500 | 5万 ~ 10万 | 10万 ~ 30万 | 30万+ | 10 ~ 30 | 当前代码经调优和集群化后较现实目标 |
| 1000 | 1000 | 10万 ~ 20万 | 20万 ~ 50万+ | 50万+ | 20 ~ 50 | 需要明显治理 |
| 2000 | 2000 | 20万 ~ 40万 | 50万 ~ 100万+ | 100万+ | 50+ | 需要平台级演进 |

说明：
- “中型客户”默认单客户用户规模约 `5万 ~ 20万`
- 如果客户以大客户为主，客户数会更少
- 如果客户以小客户为主，客户数会更多

### 6.3 2000 并发对应的用户规模怎么说
更适合的汇报口径是：

> 2000 稳定 execution 并发，大致对应 50 万到 100 万级以上 DAU 的平台能力，或者 50+ 个中型客户规模；若客户峰值更集中，则实际可承载客户数会下降。

## 7. 500 / 1000 / 2000+ 三档目标怎么理解

| 目标 execution 并发 | 当前代码可行性 | 主要矛盾 | 更准确的定性 |
|---:|---|---|---|
| 500 | 可行 | 主要是运维配套与调优 | 当前代码 + 合理运维配套可实现 |
| 1000 | 有机会 | 运维配套 + 一定代码治理 | 当前代码基础上可争取实现 |
| 2000+ | 不建议直接承诺 | 平台级架构边界 | 需要架构演进 |

### 7.1 500
- 更像“扩容 + 调优”问题
- 当前代码基础上，通过集群化与配套扩容可达成

### 7.2 1000
- 还不一定要大改架构
- 但需要更强的 MySQL/Redis/MQ/下游配套，以及一定代码治理
- 比如：热点表治理、Redis 分层准备、轻重任务治理、retry 治理

### 7.3 2000+
- 再继续上探，就不只是运维问题了
- 需要把问题上升为平台级架构演进

## 8. 2000+ 目标下，架构设计应该怎么提升

### 8.1 设计原则
- 在线执行主链路尽量轻
- 重型任务与轻型任务彻底隔离
- 强在线状态最小化
- 弱在线与分析状态异步化

### 8.2 推荐的演进方向

#### 方向一：按任务类型拆执行通道
- 轻量实时通道
- 标准异步通道
- 重型任务通道（audience / groovy / replay）

#### 方向二：把状态写入分层
- 强在线状态：执行必须依赖
- 弱在线状态：控制台秒级可见即可
- 分析/审计状态：独立承接，不与在线主表竞争

#### 方向三：Redis 分层
- 执行态 Redis：ctx / dedup / lock / quota / replay
- 缓存/路由 Redis：route / cache / pubsub
- bitmap / 大对象 Redis

#### 方向四：MQ 流量分治
- 正常流量与 retry 流量拆 topic
- 高优先级业务与低优先级业务分治
- 重型任务独立 consumer 组或独立 topic

#### 方向五：下游依赖平台化治理
- 独立连接池
- 熔断 / 限流 / 超时治理
- 容量契约化

### 8.3 一句话总结

> 2000+ 的问题，不再是“多几台机器”的问题，而是“执行模型、状态模型、缓存模型、消息模型和下游接入模型”都要升级的问题。

## 9. 2000+ 并发目标的 spec 与 plan 应该怎么写

### 9.1 推荐 spec 结构
建议新增一个专项 spec，主题聚焦在：

1. **目标定义**
   - 2000 稳定 execution 并发
   - 对应用户规模 / 客户规模
   - 定义成功标准

2. **现状边界**
   - 当前代码为什么还能到 500~1000
   - 为什么 2000+ 不能继续只靠运维放大
   - 区分代码边界与运维边界

3. **架构演进目标**
   - 执行通道拆分
   - 状态分层
   - Redis 分层
   - MQ 分治
   - 下游隔离

4. **分阶段实施路径**
   - Phase 1：先把 1000 做稳
   - Phase 2：完成 2000 的架构准备
   - Phase 3：交付 2000+ 平台能力

5. **验收标准**
   - execution 并发
   - RT / 失败率 / 积压 / DB / Redis / MQ 指标阈值

### 9.2 推荐 plan 结构
建议 implementation plan 也按阶段拆：

1. **容量基线验证**
   - 压测
   - 监控
   - 找出 500 / 1000 的真实边界

2. **配套资源升级**
   - MySQL
   - Redis
   - RocketMQ
   - 下游容量治理

3. **代码治理**
   - 热路径减 DB/Redis 往返
   - 状态写分层
   - retry/replay 治理
   - 重任务隔离

4. **架构升级**
   - 执行通道拆分
   - Redis 分层
   - 数据层治理
   - 平台级容量治理

## 10. 最终建议

### 10.1 对当前能力的表述建议
建议这样表述最准确：

> 当前代码并不是只能做到 500；500~1000 的问题，主要还是配套资源与运维实施问题，代码具备继续放大的基础。只有当目标上探到 2000+ 时，才需要明确进入平台级架构演进。

### 10.2 对后续工作的建议
1. 先把当前报告作为容量基线说明
2. 以 500 和 1000 为压测与治理目标
3. 单独立项做“2000+ 并发目标专项 spec + plan”
4. 不要在当前阶段把所有问题都表述为“代码能力不足”

### 10.3 最适合汇报的一句话

> 当前系统继续上探不是没有空间，但 500~1000 更像是“运维配套与治理”问题，真正到 2000+ 时，才需要进入平台级架构演进。
