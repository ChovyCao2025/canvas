## 第一层：运营日常痛点（每天都会碰到）

### 1. 疲劳度控制

**现状**：`FrequencyCapHandler` 存在，前端有节点配置但无全局管控UI。

**子能力拆解**：
- **全局疲劳度**：同一用户N天内最多被触达M次（跨画布、跨渠道）
- **渠道级疲劳度**：短信每天最多1条、Push每天最多3条
- **活动级疲劳度**：同一活动每周最多触发2次
- **疲劳度查询**：运营可查看某用户当前疲劳度状态（已触达几次、距上限还差几次）
- **超限策略**：超限后是跳过(silent skip)、还是排队延迟到窗口期再发

**业界标杆**：Braze 的 Rate Limiting、Iterable 的 Fatigue Rules、神策运营的触达频控。

**技术要点**：需 Redis 计数器 + 滑动窗口，`FrequencyCapHandler` 已有引擎侧逻辑，缺的是全局策略配置中心 + 前端管控面板 + 实时查询接口。

---

### 2. 效果归因

**现状**：`canvas-stats` 有执行KPI（触发/成功/失败数、节点漏斗、趋势图），无归因。

**子能力拆解**：
- **归因模型**：末次触达归因、首次触达归因、线性归因、时间衰减归因
- **归因窗口**：触达后N天内转化算有效，可配置（1h/24h/7d/30d）
- **转化事件**：可配置什么是"转化"（下单、注册、核销等），需对接 `TrackEventHandler`
- **归因报表**：按画布、按渠道、按节点的转化贡献度
- **多触点路径**：用户被3个画布触达后转化，展示完整触达链路

**业界标杆**：Braze 的 Attribution Dashboard、GrowingIO 的归因分析、神策的归因分析。

**神策对标**：神策「分析→归因分析」是独立功能模块，支持多触点归因模型。当前完全没有。

**技术要点**：核心是触达事件 + 转化事件的时序匹配。需新增归因计算引擎（可离线批处理），数据源是 `execution_trace` + 外部转化回调。

---

### 3. 模板市场

**现状**：20+ 行业模板在SQL种子数据里，无前端UI。

**子能力拆解**：
- **模板浏览**：按行业/场景/组件分类，搜索、预览、评分
- **一键复制**：从模板创建画布，自动带入节点配置和连线
- **模板贡献**：运营可把自己做好的画布发布为模板
- **版本兼容**：模板使用时自动适配当前系统的节点类型版本
- **模板统计**：使用次数、评分、最近使用

**业界标杆**：HubSpot 的 Template Marketplace、Braze 的 Canvas Templates、神策的运营模板库。

**神策对标**：神策「场景商店」是一级菜单，对应模板市场。完全是空白领域。

**技术要点**：模板本质是画布JSON的快照。需新增模板元数据表、分类标签、发布/审核流程。前端新增模板市场页面。

---

### 4. 全局静默期

**现状**：`QuietHoursHandler` 是节点级配置，每个触达节点单独设。

**子能力拆解**：
- **全局策略**：租户级默认静默期（22:00-08:00），所有触达节点自动继承
- **渠道差异**：短信严格静默、Push可配置是否遵守
- **用户偏好**：用户可自主设置免打扰时段（需 `UpdateProfileHandler` 配合）
- **时区处理**：按用户所在时区计算静默期，而非服务器时区
- **紧急突破**：营销活动可标记"紧急"忽略静默期（需审批或权限）

**业界标杆**：Braze 的 Quiet Hours、Iterable 的 Do Not Disturb、国内主流营销云均支持。

**技术要点**：全局策略存 `system_options` 表，执行时在触达Handler（SendSms/SendPush/ReachPlatform）前统一拦截。用户偏好需CDP画像支持。

---

### 5. 触达预览与试跑

**现状**：无。运营发布前无法预览效果。

**子能力拆解**：
- **内容预览**：变量插值后的消息内容实时预览（短信/Push/邮件模板+变量渲染）
- **人群估算**：配置完人群条件后，估算目标用户量级（不精确，给数量级即可）
- **Dry Run 模式**：选择少量测试用户跑完整流程，不实际触达，只看执行轨迹
- **A/B 预览**：切换不同分支看各自触达内容
- **收件人预览**：展示前N个目标用户的脱敏列表

**业界标杆**：Braze 的 Preview & Test、Iterable 的 Send Preview、HubSpot 的 Email Preview。

**技术要点**：内容预览需前端模板引擎（和后端发送时用同一套变量替换逻辑）。Dry Run 可复用现有执行引擎，在触达Handler里加一个 `dryRun=true` 标记，写轨迹但不实际发送。

## 第二层：体验升级（从能用变好用）

### 6. 画布版本管理

**现状**：无。画布只有一个版本，修改即生效（或需手动保存）。

**子能力拆解**：
- **草稿/发布版本分离**：编辑不影响线上运行版本
- **版本历史**：每次保存生成版本快照，可查看diff、回滚
- **版本对比**：并排对比两个版本的节点/连线/配置差异
- **一键回滚**：回滚到任意历史版本
- **版本备注**：保存时填写变更说明

**业界标杆**：Braze 的 Canvas Versioning、HubSpot 的 Workflow Version History。

**技术要点**：新增 `canvas_version` 表（已有但未用透），每次保存生成新版本记录，发布时切换 active_version 指针。回滚 = 激活历史版本。

---

### 7. 全局旅程视图

**现状**：无。运营只能看单个画布，无法看到某个用户被哪些画布触达过。

**子能力拆解**：
- **用户旅程时间线**：按用户ID查所有画布触达记录，按时间排列
- **触达热力图**：某用户在哪些渠道被触达了多少次
- **冲突检测**：同一时段多个画布计划触达同一用户，标红预警
- **跨画布路径**：用户从A画布流出后进入B画布的路径可视化
- **频度统计**：某用户近7/30天的触达次数和渠道分布

**业界标杆**：Braze 的 User Timeline、Iterable 的 User Event Timeline、神策的用户路径分析。

**神策对标**：神策「分析→用户路径分析」可展示跨活动的用户流转路径。旅程视图是运营视角（谁被我触达了），神策是分析视角（用户走了什么路），两者互补。

**技术要点**：核心查询是 `SELECT * FROM execution_trace WHERE user_id = ? ORDER BY created_at`。需新增聚合查询接口 + 前端时间线组件。冲突检测需比对多个画布的调度计划。

---

### 8. 智能触达时机

**现状**：`DelayHandler` 只支持固定延迟（如"延迟2小时"）。

**子能力拆解**：
- **用户活跃时段**：基于历史行为数据，每个用户有"最佳触达时段"画像
- **智能延迟**：不选"延迟2小时"，选"在用户最可能打开的时间发送"
- **渠道偏好**：某用户习惯看Push不看短信，自动降级渠道
- **时机优化**：A/B测试不同发送时间的效果，自动选最优窗口

**业界标杆**：Braze 的 Intelligent Timing、Iterable 的 Send Time Optimization。

**技术要点**：用户活跃时段需要CDP侧计算（需数据团队支持），平台侧提供"智能时机"节点类型，替代固定 Delay。最简方案：从CDP读取用户偏好时段字段，作为Delay的目标时间。

---

### 9. 画布间编排

**现状**：`SubFlowRefHandler` 支持子流程调用，但只限父子关系，无事件驱动的画布间触发。

**子能力拆解**：
- **画布触发画布**：A画布结束→自动启动B画布（事件驱动）
- **画布触发规则**：只有满足条件的用户才流入B画布
- **状态传递**：A画布的上下文变量可传递给B画布
- **画布依赖图**：可视化展示画布间触发关系
- **循环防护**：检测画布间循环触发，避免无限循环

**业界标杆**：HubSpot 的 Workflow Enrollments、Iterable 的 Journey Triggers。

**技术要点**：A画布结束时发MQ事件，B画布的 `EVENT_TRIGGER` 订阅该事件。需新增"画布完成事件"的标准化定义和MQ Topic。循环检测在画布保存时做静态分析。

---

### 10. 变量与表达式

**现状**：`ExecutionContext.flatContext` 存所有上下文，但运营配置节点时无法引用动态变量。

**子能力拆解**：
- **变量选择器**：节点配置中的输入字段可弹出变量选择面板，列出可用上下文变量
- **表达式编辑器**：支持简单表达式如 `${coupon.value * 2}` 或 `${tagger.score > 80 ? 'high' : 'low'}`
- **变量溯源**：hover变量名可看到它来自哪个节点的输出
- **类型提示**：变量有类型标注（string/number/boolean），表达式编辑器做类型检查
- **调试模式**：执行时可查看每个变量在每一步的实际值

**业界标杆**：Braze 的 Connected Content + Liquid、HubSpot 的 Personalization Tokens。

**技术要点**：变量选择器需前端解析画布DAG，上游节点的 `outputPrefix` 即可用变量空间。表达式用 SpEL 或简化版模板引擎，已有 `GroovyHandler` 可参考。调试模式需后端返回每步的 flatContext 快照。

## 第三层：规模化能力（多画布、多团队运营）

### 11. 运营日历

**现状**：无。运营靠Excel管理活动排期。

**子能力拆解**：
- **时间线视图**：甘特图/日历视图展示所有画布的运行时段
- **冲突预警**：同一时段同一人群被多个画布触达，标红提醒
- **资源占用**：展示各时段预计触达量级，预警系统容量
- **排期协同**：不同运营可在日历上看到彼此的活动安排
- **关键节点标记**：大促日、节日等标记在日历上

**业界标杆**：HubSpot 的 Campaign Calendar、Iterable 的 Campaign Schedule View。

**技术要点**：数据源是画布的调度信息（`SCHEDULED_TRIGGER` 的 cron + 预计人群量）。前端用日历组件（antd Calendar 或全量甘特图库）。冲突检测需比对多个画布的人群条件交集。

---

### 12. 数据回流闭环

**现状**：`TrackEventHandler` 可上报事件，但无消费侧——上报后数据躺在数据库里，不驱动任何自动决策。

**子能力拆解**：
- **转化事件采集**：核销、下单、注册等转化事件通过API/MQ回流
- **实时归因匹配**：转化事件自动匹配最近的触达记录
- **策略自动调整**：转化率低于阈值→自动暂停触达；高于阈值→自动放量
- **效果反馈循环**：A/B实验结果自动应用到后续活动
- **异常熔断**：投诉率/退订率异常→自动停止画布

**业界标杆**：Braze 的 Currents + Canvas Exit Criteria、Iterable 的 Journey Analytics。

**技术要点**：`GoalCheckHandler` 已有目标检查能力，缺的是"目标达成/未达成后的自动动作"。需新增画布级 exit criteria 配置 + 自动动作触发。异常熔断可复用现有 `circuit-breaker` 机制。

---

### 13. 权限细化

**现状**：只有 ADMIN / OPERATOR 两个角色，无团队/画布级权限。

**子能力拆解**：
- **团队空间**：不同业务线有独立空间，画布归属团队
- **画布级权限**：查看/编辑/发布/删除，可按人授权
- **角色模板**：运营主管、运营专员、数据分析师，权限预设
- **数据隔离**：A团队看不到B团队的画布和执行数据
- **操作审计**：谁在什么时间做了什么操作，可追溯

**业界标杆**：Braze 的 Workspace Permissions、HubSpot 的 Team Access、神策的多租户权限。

**技术要点**：RBAC → ABAC 升级。新增 `team` 表 + `canvas_team` 关联表。权限拦截可在 Controller 层 AOP 统一处理。多租户已有 `TenantController`，可复用。

---

### 14. 审批流

**现状**：`ManualApprovalHandler` 是画布内的审批节点，不是画布发布审批。

**子能力拆解**：
- **发布审批**：画布上线前需主管审批，审批通过才可激活
- **审批流程配置**：单级审批 / 多级审批 / 会签
- **审批通知**：审批待办通过飞书/企微/邮件通知审批人
- **审批意见**：审批人可填写审批意见，申请人可查看
- **紧急通道**：标记"紧急"可跳过审批（需额外权限）

**业界标杆**：神策的发布审批、HubSpot 的 Approval Workflow。

**技术要点**：画布新增 `approval_status` 字段（draft → pending → approved → rejected）。审批记录表 + 审批人配置。可对接飞书审批API。

---

### 15. 批量操作

**现状**：画布列表页只能单个操作。

**子能力拆解**：
- **批量启停**：勾选多个画布，一键启动/暂停/终止
- **批量克隆**：批量复制画布（可修改人群/渠道参数）
- **批量归档**：已结束的画布批量归档，减少列表噪音
- **批量修改**：批量修改画布的调度时间、疲劳度等参数
- **操作确认**：批量操作前二次确认 + 操作结果汇总

**业界标杆**：HubSpot 的 Bulk Actions、Braze 的 Campaign Bulk Management。

**技术要点**：前端列表页加复选框 + 批量操作栏。后端新增批量操作API（接收画布ID列表）。关键是要做幂等——已暂停的画布再暂停不报错。

---

### 16. 开放API/SDK

**现状**：`DirectCallHandler` 支持外部系统同步调用画布，但无标准化的SDK。

**子能力拆解**：
- **画布管理API**：CRUD画布、启停画布、查询状态
- **执行触发API**：程序化触发画布执行，传入自定义参数
- **事件上报API**：标准化的事件上报协议（REST + MQ）
- **查询API**：执行状态、统计结果、用户旅程
- **Webhook回调**：画布执行完成、审批通过等事件的Webhook通知
- **SDK**：Java/Python/Go SDK封装上述API

**业界标杆**：Braze REST API + SDK、Iterable API + SDK、HubSpot API。

**技术要点**：现有 `CanvasController` / `ExecutionController` 的API已较完整，缺的是标准化文档（OpenAPI spec已有）、Webhook机制、和SDK封装。Webhook需新增回调URL配置 + 事件订阅表。

## 神策对标新增缺项（与原16项不重复）

### 17. 行为分析套件

**现状**：canvas-stats 只有执行KPI和节点漏斗，无用户行为分析。

**神策对标**：神策「分析」菜单下11项分析能力，当前一个都没有。

**核心差距**：缺的不是功能模块，是数据基建——无前端埋点SDK、无OLAP引擎、事件模型太薄（event_log只有eventCode/userId/attributes，缺session_id/设备信息/页面信息等分析维度）。

**分步实施建议**：
1. 先做事件分析 + 业务漏斗（营销最需要，基于现有数据可扩展）
2. 引入OLAP后做留存 + 分布 + 路径
3. 最后做 LTV / Session / 热力等进阶分析（需前端SDK）

以下逐项展开：

---

#### 17.1 事件分析

**现状**：`TrackEventHandler` 可记录事件到 `event_log` 表，但无聚合分析能力。运营无法回答"过去7天有多少用户触发了下单事件"。

**子能力拆解**：
- **事件聚合**：按事件类型/属性维度聚合统计（计数、去重用户数、属性求和/均值）
- **时间维度**：按小时/天/周/月查看趋势
- **属性分组**：按事件属性值分组（如按"城市"看各城市下单量）
- **筛选条件**：按用户属性/事件属性筛选
- **对比分析**：环比/同比对比

**业界标杆**：神策「分析→事件分析」、GrowingIO 事件分析、Mixpanel Events。

**技术要点**：最轻量方案——在 `CanvasStatsController` 基础上新增事件聚合API，对 `event_log` 表做 GROUP BY 查询。数据量大时需引入 ClickHouse/Doris 做 OLAP。前端用 recharts（已有）做趋势图。

**可行性**：★★★★☆ — 基于现有 event_log 表即可实现基础版，无需新数据源。

#### 17.2 业务漏斗分析

**现状**：canvas-stats 有节点漏斗（DAG节点进出统计），但不是业务漏斗。运营无法分析"浏览→加购→下单"的转化率。

**子能力拆解**：
- **自定义步骤**：运营可定义漏斗步骤（不限于DAG节点，任意事件组合）
- **步骤匹配**：同一用户在时间窗口内按顺序完成各步骤
- **转化率计算**：步骤间转化率 + 整体转化率
- **漏斗对比**：不同人群/时段的漏斗对比
- **流失分析**：每步流失用户的后续行为

**业界标杆**：神策「分析→漏斗分析」、Mixpanel Funnels、Amplitude Funnel Analysis。

**技术要点**：与现有节点漏斗的区别——节点漏斗是 `execution_trace` 的 GROUP BY node_id，业务漏斗是 `event_log` 的时序窗口匹配。需新增漏斗定义表 + 漏斗计算引擎。最简方案：SQL窗口函数实现，复杂场景需OLAP。

**可行性**：★★★★☆ — 扩展现有漏斗API，数据源从 execution_trace 扩展到 event_log。

#### 17.3 留存分析

**现状**：无。运营无法回答"注册后第7天还有多少用户活跃"。

**子能力拆解**：
- **N日留存**：初始事件后第N天回访率（次日留存、7日留存、30日留存）
- **N周/月留存**：按周/月粒度计算
- **自定义留存**：初始事件和回访事件可自定义（不限于"注册→活跃"）
- **留存表格**：经典留存表格（对角线渐变色）
- **留存趋势**：留存率随时间变化的趋势图

**业界标杆**：神策「分析→留存分析」、Mixpanel Retention、Amplitude Retention。

**技术要点**：核心计算是"初始日用户集合 ∩ 第N日活跃用户集合 / 初始日用户集合"。数据量大时 MySQL 无法支撑，需 OLAP 引擎。最小可行方案：对 event_log 做定时离线计算，结果写入留存快照表。

**可行性**：★★★☆☆ — 需要OLAP或离线计算，纯MySQL方案数据量大时性能差。

**可行性**：★★★☆☆ — 与留存分析同等级，需OLAP支撑大数据量。

### 22. A/B实验管理深化

**现状**：有`AB_SPLIT`节点和`ab-experiment`页面，但实验管理能力浅。当前只有1种分流方式（画布内AB_SPLIT节点），没有试验层概念、没有结构化创建流程、没有指标管理体系。

**神策截图深度分析**（2026-05-30，基于神策A/B Testing demo界面，4张截图串联分析）：

三次截图构成一条完整的产品使用路径：试验类型选择→创建试验（4种类型）→各类型的差异化表单。串联分析后，核心洞察是——

**神策A/B Testing的本质是一个三层架构，而我们只有第三层的一个碎片**：

```
┌─────────────────────────────────────────────────────────┐
│  第一层：试验层（流量隔离空间）                              │
│  100%流量的命名空间，层内实验互斥，层间可并行                  │
│  ┌───────────────────────────────────────────────────┐  │
│  │  第二层：试验（独立实体）                              │  │
│  │  占用层的N%流量，有5种类型，4步向导创建                  │  │
│  │  ┌─────────────────┐ ┌─────────────────────────┐  │  │
│  │  │  第三层：分组      │ │  第三层：指标              │  │  │
│  │  │  对照组+实验组     │ │  检验类+观察类             │  │  │
│  │  │  参数/URL/人群     │ │  正向/反向，归因到转化事件   │  │  │
│  │  └─────────────────┘ └─────────────────────────┘  │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘

我方现状：AB_SPLIT节点 ≈ 第三层"分组"里的参数配置部分
         缺少：第一层(试验层)、第二层(试验实体)、第三层(指标体系)
```

**5种试验类型与营销画布的关联分析**：

5种试验不是平行的，而是**按"谁做实验"分成两条线**：

| 线 | 试验类型 | 实验对象 | 与画布的关系 | 营销场景 |
|----|---------|---------|------------|---------|
| **产品线** | 编程试验 | 产品功能参数(按钮颜色/排序算法) | 独立于画布，产品经理用 | 与画布无直接关联，但实验标签可被画布引用 |
| **产品线** | 可视化试验 | 页面UI(文案/布局) | 独立于画布，无需开发 | 同上 |
| **产品线** | 多链接试验 | 落地页URL对比 | 独立于画布，无需开发 | 画布触达后的落地页优化 |
| **运营线** | **多人群试验** | **不同人群的触达效果对比** | **直接嵌入画布流程** | **"高价值用户vs沉睡用户"触达对比** |
| **运营线** | **时间片轮转试验** | **长期小流量验证策略** | **画布内长期运行的一小段分支** | **新策略灰度放量** |

**关键结论：运营线的2种试验类型应优先实现，因为它们直接服务于画布编排场景。** 编程/可视化/多链接试验是产品经理的工具，与画布引擎关系不大，可后期再考虑。

**4步创建向导的分层逻辑**：

4步不是简单的线性流程，而是**从宏观到微观的逐层收敛**：

```
Step 1 基本信息    → 定义"做什么实验"（类型/名称/时长）
                      ↓
Step 2 选取受众    → 定义"在谁的流量上做"（试验层/比例/人群/分流模式）
                      ↓  ← 这里是第一层(试验层)和第二层(试验)的交汇点
Step 3 设置分组    → 定义"怎么分"（对照组+实验组，参数/URL/人群分配）
                      ↓  ← 这里是第三层(分组)的具体化
Step 4 配置指标    → 定义"怎么判断胜负"（检验类/观察类，正向/反向）
                         ← 这里是第三层(指标)的具体化
```

各试验类型在Step 2-3的分化：

| 步骤 | 编程试验 | 多人群试验 | 多链接试验 | 可视化试验 |
|------|---------|----------|----------|----------|
| Step 2 | 选取受众(层+比例+动态/静态分流) | 选择分流主体(ID类型) | 选取受众(同编程) | 选取受众(同编程) |
| Step 3 | 设置分组(参数配置Integer等) | 配置人群及进组顺序(2-5个人群) | 设置分组(URL地址) | 设置分组(预览图+编辑) |
| Step 4 | 配置指标(通用) | 配置指标(仅精准口径，无p值) | 配置指标(通用) | 配置指标(通用) |

**动态/静态分流是贯穿全流程的核心决策**，不只影响Step 2：

- 选"动态分流" → Step 3的人群会随属性变化重新分配 → Step 4不建议用标准口径指标
- 选"静态分流" → Step 3的分组固定不变 → Step 4可用任意口径指标
- 这个决策还会影响：标签生成策略、实验报告的计算方式、实验结束后的数据回溯

**子能力拆解（基于串联分析重排优先级）**：

**第一优先级：运营线实验能力（直接服务画布场景）**
- **多人群试验**：最贴近营销场景。运营圈选2-5个人群（如"高价值用户"vs"沉睡用户"），对比不同触达策略的效果。Step 3用"配置人群及进组顺序"替代通用的"设置分组"，指标仅支持精准口径。可直接嵌入画布流程——画布的SELECTOR节点选人群时，可引用多人群试验的分组结果
- **时间片轮转试验**：长期小流量验证新策略。不需要发布代码，通过时间片大小控制流量比例。适用于画布内新触达策略的灰度放量——先给1%流量跑2周，效果好再放大到10%、50%
- **试验层管理**：新增试验层概念，每个层是一个流量隔离空间（100%流量），不同实验占用层的不同比例。层内实验互斥，层间实验可并行。这是多人群试验和时间片轮转试验的基础设施——没有试验层，多实验并行时流量会冲突

**第二优先级：实验管理范式升级（结构化创建+分流模式）**
- **4步创建向导**：基本信息(名称/描述/运行时长)→选取受众(试验层/比例/人群/分流模式)→设置分组(对照+实验组/流量分配/参数)→配置指标(检验指标+观察指标)。各试验类型在Step 2-3分化：多人群试验Step 2变为"选择分流主体"、Step 3变为"配置人群及进组顺序"
- **动态/静态分流模式**：支持"随属性变更动态分流"和"分流不受属性变更影响"两种模式。这个决策贯穿全流程——动态分流下人群会重新分配、指标不建议用标准口径、标签生成策略不同、实验报告计算方式不同

**第三优先级：指标与验证体系**
- **指标管理体系**：创建指标(定义转化事件+计算方式)、添加已有指标、指标分组、指标指向(正向指标=越高越好、反向指标=越低越好)、指标属性(检验类/观察类)
- **调试设备管理**：每个分组可独立添加调试设备（用户白名单强制入组），发布前验证特定组的效果
- **实验报告深化**：统计显著性检验(p值)、效果对比(各组指标值)、推荐胜出组、置信区间

**第四优先级：闭环与协作**
- **标签自动生成**：实验上线后自动为入组用户生成标签，可配置保留天数。标签可被后续画布的人群圈选节点引用——这是实验与画布的闭环连接点
- **样本量计算器**：内置最小样本量估算工具，输入预期提升率→输出所需样本量
- **实验权限配置**：每个实验独立配置查看/编辑/发布权限
- **进组事件标准化**：定义`$ABTestTrigger`事件，外部系统可订阅

**神策对标**：神策「A/B测试」是独立一级菜单，包含试验列表、试验层、指标管理、调试设备等子模块。5种试验类型按使用角色分两条线——产品线(编程/可视化/多链接)和运营线(多人群/时间片轮转)。我方当前AB_SPLIT只是画布内一个节点，缺少整个实验管理层。优先实现运营线2种类型，它们直接服务于画布编排场景。

**技术要点**：

1. **试验层**：新增 `experiment_layer` 表（layer_id, name, total_traffic_pct, status）。每个层代表100%流量空间，实验在层内按比例分配。层间流量可叠加（用户可同时在不同层的实验中）。这是运营线实验的基础设施
2. **实验独立表**：新增 `ab_experiment` 表（experiment_id, name, type, layer_id, traffic_pct, audience_mode, split_mode, status, created_by, created_at）。type字段区分多人群/时间片轮转等；split_mode字段记录动态/静态分流。实验不再依附于画布节点，而是独立实体
3. **分流引擎升级**：当前AB_SPLIT的分流逻辑迁移到独立分流服务，支持动态/静态两种模式。动态模式需在用户属性变更时重新计算分组（可复用CDP的实时标签更新机制）。分流结果通过`$ABTestTrigger`事件通知画布引擎
4. **指标对接**：实验指标需对接 `TrackEventHandler` + `GoalCheckHandler` + 归因(#2)。检验类指标做显著性计算，观察类指标做趋势展示。多人群试验仅支持精准口径指标
5. **标签生成**：实验上线→异步生成用户标签（批量写入CDP标签表）。保留天数到期后自动清理。标签是实验与画布的闭环连接点——画布的SELECTOR/TAGGER节点可引用实验标签
6. **前端改造**：`ab-experiment` 页面从简单列表升级为：试验类型选择页(5种卡片) + 4步创建向导(按试验类型分化Step 2-3) + 试验层管理 + 指标管理 + 调试设备管理。与画布编辑器联动——创建实验后可在画布中引用实验分组

### 23. 自定义查询

**现状**：无。运营无法自助查数据，必须找开发写SQL。

**子能力拆解**：
- **SQL查询**：运营可写SQL查数据（需权限控制和资源限制）
- **可视化查询**：拖拽式查询构建器，无需写SQL
- **查询保存**：常用查询保存为书签
- **查询结果下载**：导出CSV/Excel
- **查询审计**：记录谁查了什么数据

**神策对标**：神策「分析→经营分析→自定义查询」。

**技术要点**：SQL查询需沙箱化（只读、超时限制、行数限制）。可视化查询可参考 react-querybuilder（项目已依赖）。

## 第四层：营销系统核心能力（合规风控→权益激励→预算ROI→生命周期）

> 前三层聚焦"功能缺项"（疲劳度、归因、模板市场等），第四层补充营销系统作为商业产品必须具备的核心能力维度——合规风控、权益激励、预算管理、生命周期运营。这些不是锦上添花，而是营销平台从"能用"到"可信、可控、可持续"的关键跨越。

---

### 24. 营销合规审核

**现状**：无。触达内容无合规审核流程，存在法律风险。

**子能力拆解**：
- **内容审核**：触达内容上线前需审核（敏感词检测、合规检查）
- **审核流程**：单级/多级审核，审核意见记录
- **敏感词库**：维护敏感词库，自动检测内容中的敏感词（支持正则匹配）
- **审核通知**：审核待办通知审核人（对接飞书/企微）
- **审核历史**：记录审核历史，可追溯

**业界标杆**：中国营销平台标配（神策、Convertlab、火山引擎均有）；Braze/Iterable无原生审核（海外合规靠授权管理替代）。

**为什么重要**：中国广告法合规要求——虚假宣传、绝对化用语（"最好""第一"）、敏感词均需拦截。无审核 = 违规风险。

**技术要点**：新增 `content_audit` 表（content_id, status, auditor, audit_at, comment）。敏感词检测：正则匹配或对接第三方API。审核通过后才能发布画布。与 #14 审批流 共享审核基础设施。

---

### 25. 用户隐私保护（GDPR/PIPL）

**现状**：无。无用户授权管理、无数据删除机制。

**子能力拆解**：
- **授权管理**：记录用户对各渠道的授权状态（短信授权、Push授权、邮件授权），未授权用户不触达
- **授权状态查询**：查询用户对各渠道的授权状态
- **授权变更**：用户可随时变更授权状态（自助或API）
- **数据删除**：用户请求数据删除时，级联删除所有相关数据（GDPR Right to be Forgotten / PIPL 删除权）
- **数据导出**：用户请求数据导出（GDPR Data Portability / PIPL 可携带权）

**业界标杆**：Braze Subscription Groups + Data Deletion；Iterable Subscription Preferences；HubSpot GDPR Compliance Toolkit。

**中国PIPL合规要求**（《个人信息保护法》）：
1. **明示同意**：收集用户信息前需明示告知并获得同意
2. **目的限制**：只能为告知的目的使用用户信息
3. **最小必要**：只收集必要的信息
4. **用户权利**：用户有权查询、更正、删除其个人信息
5. **撤回同意**：用户可随时撤回授权

**技术要点**：新增 `user_consent` 表（user_id, channel, consent_status, consent_at, source）。触达Handler执行前检查授权状态，未授权则跳过。提供数据删除API（级联删除 `execution_trace` + `event_log` + CDP数据）。提供数据导出API。

---

### 26. 退订/黑名单管理

**现状**：无。无退订机制，无黑名单管理。

**子能力拆解**：
- **退订入口**：短信/邮件底部带退订链接，用户点击后不再触达
- **退订状态管理**：记录用户退订状态（全局退订/渠道退订/某画布退订）
- **黑名单管理**：运营可手动添加黑名单用户，黑名单用户永不触达
- **黑名单导入**：批量导入黑名单用户（CSV）
- **退订/黑名单查询**：查询用户是否在退订/黑名单中

**业界标杆**：Braze Unsubscribe + Subscription Groups；Iterable Unsubscribe。中国法规要求：用户拒绝后不得再发。

**技术要点**：新增 `user_blacklist` 表（user_id, reason, added_by, added_at）+ `user_unsubscribe` 表（user_id, channel, canvas_id, unsubscribed_at）。触达Handler执行前检查退订/黑名单状态。短信/邮件模板增加退订链接占位符 `${unsubscribe_url}`。

---

### 27. 频次合规管控

**现状**：`FrequencyCapHandler` 有业务频控，但无合规层面的频次管控。

**子能力拆解**：
- **法规频次限制**：中国法规要求——营销短信每天最多1条（用户明确同意除外）
- **平台频次限制**：各渠道平台限制（微信每天1条模板消息、Push各平台限制）
- **频次合规检查**：触达前检查是否违反法规/平台频次限制
- **频次合规报告**：生成频次合规报告，供监管审查

**业界标杆**：Braze Rate Limiting；Iterable Fatigue Rules。中国法规：《通信短信息服务管理规定》要求未经同意每天最多1条商业性短信息。

**技术要点**：在现有 `FrequencyCapHandler` 基础上增加合规频次检查层。合规频次限制优先级高于业务频次限制。超限时自动跳过或延迟到合规窗口期。合规频次配置存 `system_options` 表。

---

### 28. 优惠券管理系统

**现状**：`CouponHandler` 可在画布中触发优惠券发放，但无优惠券生命周期管理（创建、发放、核销追踪）。

**子能力拆解**：
- **优惠券创建**：创建不同类型优惠券（满减、折扣、立减、兑换券），配置规则（金额、门槛、有效期、适用商品）
- **券码生成**：批量生成唯一券码，支持自定义格式（前缀+随机码+校验位）
- **发放策略**：按人群发放、按行为触发发放、定时批量发放、限量发放（先到先得）
- **核销追踪**：记录券码使用状态（未使用/已使用/已过期），对接业务系统核销回调
- **ROI分析**：优惠券带来的GMV、核销率、ROI计算

**业界标杆**：Braze/Iterable/HubSpot 无原生优惠券管理，通过 Connected Content/API 对接外部系统。Salesforce Commerce Cloud 有完整优惠券生命周期管理。

**技术要点**：新增 `coupon` 表（coupon_id, name, type, value, threshold, valid_from, valid_to, total_quantity, used_quantity）+ `coupon_code` 表（code, coupon_id, user_id, status, issued_at, used_at）。扩展 `CouponHandler` 支持从 coupon 表发放。核销回调API：`POST /api/coupon/redeem`。

### 29. 积分/会员体系引擎

**现状**：`PointsOperationHandler` 可在画布中触发积分操作，但无积分账户管理和会员等级体系。

**子能力拆解**：
- **积分规则引擎**：定义积分获取规则（消费1元=1积分、签到+5积分、邀请好友+50积分）
- **积分账户管理**：用户积分余额、积分流水、积分过期管理
- **会员等级体系**：定义等级规则（积分达到1000升级银卡）、等级权益配置
- **积分消耗**：积分兑换优惠券、积分抵扣、积分商城
- **积分激励画布**：在画布中配置"完成某行为+X积分"节点

**业界标杆**：Braze/Iterable 无原生积分/会员体系。Salesforce 有 Loyalty Management 模块。中国平台（神策/Convertlab/火山引擎）均无原生积分模块，需对接业务系统。

**技术要点**：积分系统通常独立于营销平台，营销平台只负责"触发积分发放"。可扩展 `PointsOperationHandler` 支持积分字段更新，或对接外部积分系统API。会员等级可基于CDP标签实现（等级=标签值）。

---

### 30. 活动预算管理

**现状**：无。运营无法在平台内管理活动预算，无法控制触达成本。

**子能力拆解**：
- **预算设置**：为画布/计划设置预算上限（总预算、日预算、渠道预算）
- **成本追踪**：实时追踪触达成本（短信费用=发送量×单价、Push费用、邮件费用）
- **预算消耗预警**：预算消耗达80%/90%时告警通知
- **预算超限熔断**：预算耗尽自动暂停画布，防止超支
- **预算审批**：大额预算需审批后才能执行

**业界标杆**：HubSpot Campaign Budget；Salesforce Marketing Cloud Budget。Braze/Iterable 无原生预算管理。

**技术要点**：新增 `canvas_budget` 表（canvas_id, total_budget, daily_budget, used_budget, channel_budgets JSON）。触达Handler执行前检查预算余额，不足时返回 `NodeResult.terminal()` 并触发画布暂停。渠道单价配置存 `channel_cost` 表（channel, unit_cost, effective_date）。

---

### 31. 渠道成本追踪

**现状**：无。不知道短信发了多少条、花了多少钱。

**子能力拆解**：
- **渠道单价配置**：配置各渠道单价（短信0.05元/条、Push免费、邮件0.001元/封）
- **实时成本计算**：发送量 × 单价 = 成本
- **成本趋势分析**：按日/周/月查看各渠道成本趋势
- **成本对比**：不同画布、不同渠道的成本对比
- **成本预警**：成本异常波动时告警

**技术要点**：新增 `channel_cost` 表（channel, unit_cost, effective_date）。在 `execution_trace` 表增加 `cost` 字段（DECIMAL）。触达Handler执行后写入成本。定时任务聚合计算成本趋势。

---

### 32. ROI/ROAS计算

**现状**：无。无法衡量营销活动的投入产出比。

**子能力拆解**：
- **ROI计算**：(收入 - 成本) / 成本 × 100%
- **ROAS计算**：收入 / 成本（广告投入产出比）
- **归因ROI**：结合归因模型，计算各触达点的ROI贡献
- **ROI趋势**：按日/周/月查看ROI趋势
- **ROI对比**：不同画布、不同渠道、不同人群的ROI对比

**业界标杆**：Braze Attribution Dashboard；HubSpot Revenue Attribution；Salesforce Marketing Cloud ROI Reports。

**技术要点**：需对接订单/支付系统获取收入数据，与触达记录关联（依赖 #2 归因）。最简方案：手动导入收入数据，与 execution_trace 按用户+时间窗口关联。

---

### 33. 用户生命周期阶段识别

**现状**：CDP有标签能力，但无标准化的生命周期阶段定义和自动识别。

**子能力拆解**：
- **阶段定义**：定义生命周期阶段（新用户→活跃→成熟→衰退→流失）
- **阶段规则**：定义阶段转换规则（注册后7天无活跃→流失预警）
- **自动识别**：基于用户行为自动计算当前阶段
- **阶段可视化**：展示用户在各阶段的分布（漏斗图/桑基图）
- **阶段趋势**：用户阶段迁移趋势分析

**业界标杆**：Braze Lifecycle Segments；Iterable Lifecycle；HubSpot Lifecycle Stage（核心CRM概念：Subscriber→Lead→MQL→SQL→Customer→Evangelist）。

**生命周期 vs 画布/旅程**：

| 维度 | 生命周期 | 画布/旅程 |
|------|---------|---------|
| 视角 | 用户视角（用户当前在哪个阶段） | 运营视角（我要对用户做什么） |
| 时间维度 | 长期（用户整个生命周期） | 短期（一次活动/旅程） |
| 触发方式 | 用户行为自动触发阶段转换 | 运营主动触发或事件触发 |
| 关系 | 生命周期阶段可作为画布的触发条件/筛选条件 | 画布执行结果可影响用户生命周期阶段 |

**技术要点**：新增 `lifecycle_stage` 表（stage_id, name, rules JSON, display_order）。在CDP用户属性中增加 `lifecycle_stage` 字段。定时任务（ElasticJob）计算用户阶段（基于行为数据）。画布 SELECTOR 节点可按生命周期阶段筛选用户。

---

### 34. 生命周期自动化编排

**现状**：无。运营需要手动为不同阶段用户创建画布。

**子能力拆解**：
- **阶段触发画布**：用户进入某阶段时自动触发对应画布（新用户→新手引导画布）
- **阶段转换触发**：用户从A阶段转换到B阶段时触发画布（活跃→衰退→召回画布）
- **阶段策略模板**：为每个阶段预设触达策略模板
- **阶段效果追踪**：追踪各阶段用户的转化效果
- **阶段策略优化**：基于效果数据优化阶段策略

**技术要点**：生命周期阶段变更时发MQ事件（`LifecycleStageChanged`）。画布的 `EVENT_TRIGGER` 订阅阶段变更事件。或新增 `LIFECYCLE_TRIGGER` 触发器类型。

---

### 35. 阶段转换触发器

**现状**：无。

**子能力拆解**：
- **转换规则定义**：定义阶段转换条件（连续7天未活跃→流失预警）
- **转换事件**：阶段转换时生成标准化事件（`LifecycleStageChanged`）
- **转换触发画布**：转换事件自动触发对应画布
- **转换日志**：记录用户阶段转换历史

**技术要点**：定时任务检测阶段转换。转换时生成事件并发布到MQ。画布订阅事件触发执行。与 #9 画布间编排 共享事件驱动基础设施。

## 第五层：代码库深度扫描发现的缺项（#36–#64）

以下29项是原文档35项未覆盖的缺项，来源于代码库逐文件扫描（Handler实现、Controller/Service、Flyway迁移、前端页面、TODO/FIXME注释）。

### 5.1 引擎/合规关键缺口（P0）

#### 36. MarketingPolicyService 未接入发送链路

**现状**：`MarketingPolicyService` 已实现 `consentAllowed()`, `suppressionAllowed()`, `quietHoursAllowed()`, `channelAvailable()`, `consumeFrequency()` 五个策略方法，但 `AbstractSendMessageHandler` 完全没有调用。所有 SendXxxHandler 发送前不做任何合规检查。

**子能力拆解**：
- **策略检查链**：consent → suppression → quietHours → channelAvailable → frequencyCap，任一不通过返回 `NodeResult.suppressed()`
- **原因码透传**：suppressed 结果需携带具体原因（consent_denied / in_suppression / quiet_hours / channel_unavailable / frequency_exceeded）
- **日志记录**：每次策略拦截写入 execution_trace

**技术要点**：修改 `AbstractSendMessageHandler.executeAsync()`，在模板解析后、投递前插入策略检查链。`MarketingPolicyService` 已存在，只需注入并调用。

**涉及文件**：`AbstractSendMessageHandler.java`

#### 37. event_log 缺少分析维度字段

**现状**：`event_log` 表仅有 `event_code`, `user_id`, `attributes`(JSON), `canvas_triggered`, `canvas_count`，无法支撑任何有意义的分析（漏斗、留存、路径、归因均依赖事件维度）。

**子能力拆解**：
- **Session维度**：`session_id` — 会话标识，Session分析必需
- **设备维度**：`device_id`, `platform` — 跨设备分析、App点击分析必需
- **页面维度**：`page_url`, `referrer` — 路径分析、热力图必需
- **网络维度**：`ip`, `user_agent` — 地域分析、设备识别必需
- **时间精度**：`event_time` DATETIME — 精确到秒的事件时间（当前只有 created_at）
- **版本管理**：`event_version` INT — 事件Schema版本

**技术要点**：Flyway V89 ALTER TABLE 增加字段；`TrackEventHandler` 从 ExecutionContext 提取维度写入；`EventTriggerHandler` 将触发事件携带的维度透传。

**涉及文件**：`V89__event_log_analysis_dimensions.sql`, `TrackEventHandler.java`, `EventTriggerHandler.java`

#### 38. CanvasExecutionService dedup TTL 修复

### 关键洞察

1. **AI是最大盲区**：64项中仅1项AI（#63 画布内LLM节点），业界已有10+AI能力（预测分析、自主Agent、内容生成）
2. **预测 vs 描述**：64项聚焦描述性分析（发生了什么），业界已转向预测性分析（将发生什么）——流失预测、LTV预测、下次购买日期
3. **客户面功能完全缺失**：64项全部面向运营人员，业界已有面向用户的自助门户、偏好中心、内置客服
4. **新渠道缺失**：RCS富媒体、社交广告直投、AEO(AI搜索)追踪均为新渠道类型
5. **营销=销售割裂**：64项纯营销视角，业界已打通CRM、线索评分、销售序列自动化
6. **信任体系缺失**：HIPAA、SOC 2认证成为企业采购门槛
