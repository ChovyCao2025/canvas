# 2000+ Execution 并发目标专项 Spec

## 1. 背景

当前系统已经具备继续从 500 向 1000 execution 并发上探的基础能力，但这一区间的主要问题仍然集中在：
- MySQL / Redis / RocketMQ / 下游服务的配套资源是否足够
- 压测、监控、调优是否到位
- 热点治理与重任务治理是否完成

当目标进一步上探到 **2000+ 稳定 execution 并发** 时，单纯依赖扩应用和扩基础设施，收益会开始明显下降。此时需要从“容量扩容问题”上升到“平台级架构演进问题”。

---

## 2. 目标定义

### 2.1 总目标
建设一套支持 **2000+ 稳定 execution 并发** 的平台能力。

### 2.2 目标口径
- execution 并发 = 高峰时刻同时执行的画布执行数
- 不等于 HTTP 连接数
- 不等于总客户数

### 2.3 对应用户规模口径
2000 execution 并发大致对应：
- **50万 ~ 100万+ DAU（中性口径）**
- **50+ 中型客户规模（粗估）**

若客户峰值更集中，则可承载客户数会更低。

---

## 3. 当前边界

### 3.1 当前代码仍可继续上探的原因
- WebFlux + Disruptor + MQ 已具备一定削峰和分发能力
- 虚拟线程对部分阻塞任务有缓冲作用
- 当前代码已经具备 execution 并发准入、overflow/retry、Redis 限流等基础治理能力

### 3.2 为什么到 2000+ 不能继续只靠运维放大
当 execution 并发到 2000+ 时，系统主要矛盾会从“资源不够”转成“架构边界不清”：
- 在线执行状态写入仍然较重
- Redis 职责过多
- 轻重任务共享过多资源
- retry / replay / audience / groovy 仍未彻底分治
- 下游慢调用会更容易传染主链路

结论：
- 500~1000 更多是运维配套与治理问题
- 2000+ 需要平台级架构演进

---

## 4. 架构演进目标

### 4.1 执行通道拆分
将执行层拆成至少 3 类通道：
1. 轻量实时通道
2. 标准异步通道
3. 重型任务通道

目标：
- 轻任务不被重任务拖垮
- 集群能够按不同任务类型独立扩容

### 4.2 状态模型分层
把状态拆成：
1. 强在线状态
2. 弱在线状态
3. 分析/审计状态

目标：
- 减轻 MySQL 热表压力
- 降低同步写放大

### 4.3 Redis 分层
至少拆为：
1. 执行态 Redis
2. 缓存/路由 Redis
3. bitmap / 大对象 Redis

目标：
- 保证在线执行低延迟
- 避免大对象影响主链路

### 4.4 MQ 流量分治
按用途拆分：
- 正常流量 topic
- retry / overflow topic
- 重型任务 topic

目标：
- 高峰期先保主链路
- 避免 retry 洪峰冲击正常流量

### 4.5 下游依赖平台化治理
- 按依赖类型独立连接池
- 熔断 / 限流 / 超时治理
- 建立容量契约

目标：
- 减少下游长尾放大效应

---

## 5. 分阶段实施路径

### Phase 1：把 1000 做稳
目标：
- 通过压测和调优，把 500~1000 区间边界做清楚并稳定住

### Phase 2：完成 2000 的架构准备
目标：
- 完成执行通道拆分设计
- 完成状态分层设计
- 完成 Redis 分层设计
- 完成 MQ 分治设计

### Phase 3：交付 2000+ 平台能力
目标：
- 基于新架构完成实现、验证和上线

---

## 6. 验收标准

### 6.1 性能指标
- 稳定 execution 并发 ≥ 2000
- 核心链路 RT 可控
- 错误率低于约定阈值
- retry / overflow / backlog 在可控范围

### 6.2 基础设施指标
- MySQL 连接、慢 SQL、热点表压力可控
- Redis RT、热点 key、连接数可控
- RocketMQ 无持续积压
- 下游依赖长尾受控

### 6.3 架构指标
- 轻任务与重任务已隔离
- 状态分层完成
- Redis 分层完成
- MQ 正常流量与 retry 流量隔离完成

---

2. 分阶段实施 plan
3. 压测与验收基线
4. 上线与回滚策略
