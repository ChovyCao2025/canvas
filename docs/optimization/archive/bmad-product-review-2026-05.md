# BMAD 产品设计合理性审查 — 完整报告

> 审查日期：2026-05-31
> 审查方法：BMAD Master 三步审查（PM Checklist + 竞品分析 + UI/UX Spec）+ 8轮深度扫描
> 审查范围：产品经理视角、UI/UX设计视角、API设计、数据模型、运营工作流、安全合规、CDP/人群、通知协作、集成扩展、错误恢复UX、移动/i18n
> 累计发现：约208项产品缺项（含二次审核28项新增）

---

## 总判定：能用，但离"好用"差距显著

**核心优势**：DAG画布编排能力与Braze/Iterable基本持平，60+节点Handler，Schema驱动动态表单，配置与执行分离，防资损设计。

**核心差距**：运营管控层（归因/疲劳度/模板/旅程/预算）全面落后于所有标杆。8轮深度扫描发现约180项产品缺项。

## 四、API设计缺项

### P0级

| 缺项 | 影响 |
|------|------|
| **无Webhook/事件订阅** | 营销自动化无出站集成，无法连接CRM/分析平台/Slack |
| **无画布导出/导入** | 无法跨环境迁移(staging→prod)，无灾难恢复 |
| **无批量操作API** | 20+画布逐个发布/下线/归档 |
| **无执行报告导出(CSV/Excel)** | 无法分享给无仪表盘权限的利益相关者 |
| **多个列表端点无分页** | CDP用户/画布用户/管理员用户——数据增长后崩溃 |

### P1级

| 缺项 | 影响 |
|------|------|
| 无强制Kill所有执行API | 紧急止损无按钮 |
| 画布搜索仅name+status | 无法按日期/创建者/业务线筛选 |
| 执行端点无速率限制 | 直调/行为触发完全裸露 |
| 5种路径前缀混乱(/canvas/,/admin/,/auth/,/meta/,/ops/) | API学习成本高 |
| 无API版本策略 | 破坏性变更影响所有消费者 |
| 无System Options的CREATE/DELETE | 无法增删配置项 |
| 无Template的UPDATE/DELETE | 模板只能创建不能编辑删除 |
| 无Admin User的enable端点 | 禁用用户无法重新激活 |
| 无数据源连接测试端点 | 必须保存后试查询才发现凭据错误 |

## 五、数据模型缺项

### CRITICAL

| 缺项 | 影响 |
|------|------|
| **无软删除** | 删除操作永久不可恢复，支持表（API/事件/标签定义等）全部硬删 |
| **无数据归档/保留策略** | execution_trace/event_log/message_send_record无限增长 |

| **无GDPR删除权实现** | 无用户级级联删除，8张PII表无法原子清理 |

### HIGH

| 缺项 | 影响 |
|------|------|
| 零外键约束 | 数据完整性全靠应用逻辑，删canvas留孤儿记录 |

## 七、安全合规缺项

### CRITICAL (4项)

| 缺项 | 法规依据 |
|------|---------|
| **无删除权API或级联删除** | GDPR Art.17, PIPL Art.47 |
| **无审计日志基础设施** | GDPR Art.5(2)/24/30, PIPL Art.55 |
| **/ops/** 端点无认证公开可访问 | GDPR Art.32, PIPL Art.51 |
| **无跨境数据传输控制** | GDPR Ch.V, PIPL Art.38-43 |

### HIGH (5项)

| 缺项 | 法规依据 |
|------|---------|
| 无公开consent/unsubscribe端点给终端用户 | GDPR Art.7, CAN-SPAM s.5, CASL s.6-7/11 |
| PII字段无加密存储 | GDPR Art.32(1)(a), PIPL Art.51 |
| 无数据保留/自动清理策略 | GDPR Art.5(1)(e), PIPL Art.6 |
| Groovy evaluateExpression()绕过沙箱 | GDPR Art.32, PIPL Art.51 |
| /canvas/execute/direct/* 和 /canvas/trigger/behavior 公开可访问 | GDPR Art.32 |

### MEDIUM (6项)

| 缺项 | 说明 |
|------|------|
| PII脱敏未应用于应用日志 | DataMaskingUtil仅2处调用 |
| 仅4个粗粒度角色，无资源级权限 | 无画布级/节点级/操作级权限 |
| CORS默认允许所有来源 | `canvas.cors.allowed-origins: "*"` |
| Swagger UI和Actuator details生产暴露 | springdoc/swagger-ui.enabled: true |
| DataSource密码明文存储 | 与用户密码不同，这是第三方系统凭据 |
| 事件上报密钥有硬编码默认值 | `canvas-event-report-secret-2026!!` |

## 十一、错误恢复UX缺项

| 缺项 | 优先级 | 影响 |
|------|--------|------|
| **无DLQ管理UI** | P0 | 操作员无法浏览/检查/重放死信，必须用curl |
| **无失败通知** | P0 | 画布执行失败时操作员无推送通知，可能数小时不知 |
| **无熔断器可见性** | P0 | 操作员看不到哪些熔断器打开，无法诊断"为什么所有API_CALL节点失败" |
| 无单节点重试或跳过继续 | P1 | 只能完整重放整个执行 |
| 无部分执行状态 | P1 | 执行只有SUCCESS/FAILED，无"3/5分支成功" |
| 无错误分类体系 | P1 | 原始错误字符串，无errorCategory字段 |
| 重启后RUNNING孤儿无清理 | P1 | 崩溃前RUNNING的执行记录永远保持RUNNING |

## 十三、二次审核 — 业务逻辑盲区（7项新发现）

### 13.1 发布前无受众/成本估算

操作员发布画布前无法预览触发量级。`TriggerPreCheckService`仅在执行时扣减配额，不提供预发布投影。发布"所有购买用户"这类宽触发画布时，操作员无法预见执行量或成本。

### 13.2 MQ Topic存在性不验证

`CanvasService.registerTriggerRoutes()`无条件注册路由到Redis，不验证MQ Topic是否存在于消息代理。拼写错误或未创建的Topic导致画布发布成功但永远不触发——**静默失败**，无错误/指标/告警。

### 13.3 跨画布用户去重/优先级协调缺失

两个画布重叠触发条件可同时执行同一用户，零协调。无画布间优先级方案，无法表达"画布A应优先于画布B"。

### 13.4 执行期间用户画像不快照

`DagEngine.resolveConfig()`每次节点求值动态读取ExecutionContext。若CDP画像在执行中途被其他画布修改，同一执行内的后续节点看到更新后的画像，产生内部不一致行为。

### 13.5 无总节点数上限

`MAX_NODE_DEPTH=200`限制深度但不限制宽度。500节点扇出画布合法，结合无单画布并发执行限制，高触发量可消耗无上限集群资源。

### 13.6 GOTO节点无迭代上限——可能无限循环

`LoopHandler`有`maxIterations`保护，但`DagEngine.resetReachableUntilSource()`处理GOTO节点时无迭代计数器。GOTO指向上游节点创建循环，仅靠全局超时(600s)兜底。

### 13.7 循环子画布链检测不完整

`CanvasService.validateSubFlowDependencies()`仅检查直接自引用，不检测传递环(A→B→C→A)。运行时直到`MAX_NODE_DEPTH=200`才报错，错误信息不提示循环子画布链。

## 十七、二次审核 — 19个业务场景可行性审计

### 完全不可能（10个）

| 场景 | 为什么不可能 |
|------|-------------|
| A/B自动推广胜者 | 无转化追踪/统计显著性/自动推广机制 |

### 勉强可行但痛苦（6个）

| 场景 | 缺什么 |
|------|--------|
| 7天欢迎系列 | DELAY链可模拟但无drip抽象/无天单位延迟 |
| 组合条件(购物车放弃+本周无消息) | IF_CONDITION+FREQUENCY_CAP链可工作但无统一条件构建器 |
| 模板克隆+变量替换 | clone存在但无模板变量系统 |

| 生命周期阶段逻辑 | lifecycleStage字段存在但无原生生命周期路由节点 |
| 地域定向 | region字段存在但无原生geo-target节点 |

### 根因模式

平台被构建为**流程执行引擎**，而非**营销运营平台**。三个系统性差距反复出现：

1. **无价值层** — 追踪"是否发生"(执行计数/成功失败)但从不追踪"价值多少"(收入/转化/ROI) → 5个场景不可能
2. **无跨画布感知** — 每个画布是孤岛，无平台级跨画布查询 → 4个场景不可能
3. **无高层营销抽象** — 提供底层构建块(IF/DELAY/WAIT)但无组合模式(drip/cascade/lifecycle/VIP) → 3个不可能 + 6个痛苦手动
