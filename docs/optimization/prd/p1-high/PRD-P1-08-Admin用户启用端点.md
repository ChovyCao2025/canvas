## 1. 问题描述

### 1.1 现状

- 用户状态通过 Feature Gate 控制，禁用后无法重新启用
- 没有提供专门的 API 端点用于启用/禁用管理员账户
- 封禁后恢复属于人工干预，流程长且无日志

### 1.2 痛点

- 用户账号封禁误操作后恢复困难
- 切换租户/环境时需要来回开关禁用状态
- 审计链路不完整，无法追踪用户状态变更历史

### 1.3 竞品对标

| 竞品 | 能力 |
|------|------|
| Iterable | User Retire/Update Locked User, 需动员审批 |
| Braze | User Deactivate/Reactivate, 支持恢复说明 |

---

## 2. 目标与价值

### 2.1 用户故事

> 作为 系统管理员，我希望 根据用户 ID 查询并启用/禁用管理员账户，以便 在误操作时快速恢复，在需要隔离时完成管控。

### 2.2 成功指标

- 状态切换在 500ms 内完成
- 所有变更记录在日志中且支持第三方检索
- 重启站点不影响已有用户状态变更

### 2.3 不做会怎样

- 状态不可逆且不记录审计，给公司带来合规风险
- 无法针对租户或部门进行精细划分

---

## 3. 功能需求

### 3.1 核心功能

1. **用户状态查询** — GET `/api/v3/admin/users/{id}` 返回当前状态
2. **启 用终端** — PUT `/api/v3/admin/users/{id}/enable` 恢复被禁止状态
3. **禁 用终端** — PUT `/api/v3/admin/users/{id}/disable` 激活限制状态
4. **状态变更审计** — 记录变更时间、操作人、原状态、新状态

### 3.2 详细描述

- API 响应：
  ```json
  {
    "userId": "string",
    "status": "ACTIVE|DISABLED|SUSPENDED",
    "lastStatusChangeAt": "RFC3339",
    "lastModifiedBy": "string"
  }
  ```

- 权限控制：只有 ADMIN 角色可启用/禁用，启用后立即生效

### 3.3 交互流程

1. Admin 在管理界面搜索用户 ID
2. 点击「启用」或「禁用」，输入原因
3. 系统更新状态并记录审计日志
4. 用户重新登录时校验状态，若禁用则拒绝

---

## 4. 非功能需求

- **性能要求**: 启用/禁用必须避免 DB 上下文切换延迟，只更新状态字段
- **安全要求**: 禁用前应检查该用户是否有进行中的关键任务，提示影响
- **可用性要求**: 变更后需通知相关团队（可用邮件系统）

---

## 5. 验收标准

- [ ] 禁用/启用 API 端点 200 OK & status 正确更新
- [ ] 被禁用用户无法登录，且拒绝理由清晰
- [ ] 审计日志记录完整（时间戳、操作人、原/新状态）
- [ ] 禁用操作时不甩卸进行中的任务（避免掉单）

---

## 6. 技术建议

### 6.1 涉及模块

- 后端 AdminController
- 审计模块（ClientSide/ServerSide）

### 6.2 技术要点

- 使用 MyBatis-Plus 的 `LambdaUpdateWrapper` 更新用户状态
- 采用 Event-Driven 通知变更（可选邮件/IM 通知）
- 状态变更存储于审计表，不修改数据库约束

### 6.3 预估工作量

- API 开发：2 人天
- 前端界面：2 人天
- 审计日志：1 人天

## 7. 依赖与风险

### 7.1 前置依赖

- 现有 User/Account 模型结构

### 7.2 风险

- 禁用账号后进行中的执行可能抛异常
- 审计策略缺失会导致责任链不完整
