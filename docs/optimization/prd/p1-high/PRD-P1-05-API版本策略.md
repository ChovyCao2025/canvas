## 1. 问题描述

### 1.1 现状

- 当前 API 路径未显式包含版本号（如 `/api/v3` 或 `/v1`）
- 破坏性变更（如参数增删、响应结构修改）直接发布会影响所有消费者
- 缺乏版本规划和向上兼容机制

### 1.2 痛点

- 端到端集成方难以适配增量变更，反复升级成本高
- 生产环境升级时存在发布回滚风险
- 无法多做同时并行不同版本功能（如旧版稳定、新版实验）

### 1.3 竞品对标

| 竞品 | 能力 |
|------|------|
| Iterable | 固定 `/api/v3`，变更通过修改字段/添加新字段，避免破坏性变更 |
| Braze | 维护 `/kg/v1` 与 `/kg/v2` 并存，旧版支持一定过渡期 |
| Stripe | `/v1/{resource}`，严格向后兼容，新增字段可选 |

---

## 2. 目标与价值

### 2.1 用户故事

> 作为 API 提供方和消费者，我希望 通过版本化策略和兼容性规范，实现 API 的平稳迭代与向后兼容，防止破坏性变更影响现有集成。

### 2.2 成功指标

- 90% 以上 API 变更为向后兼容（新增字段/可选参数）
- 破坏性变更有明确标记和过渡期（30 天兼容层或内网分支不强制）
- API 文档中清晰标注版本兼容层级

### 2.3 不做会怎样

- 破坏性变更导致下游适配灾难，难以回退
- 统一升级时沟通成本高，依赖人工指引
- 无法进行灰度实验和新特性并行接入

---

## 3. 功能需求

### 3.1 核心功能

1. **版本显式化** — 所有 API 路径包含版本号（如 `/api/v3`）
2. **兼容性规范** — 定义破坏性变更的准入/冲突规则
3. **过渡期支持** — 破坏性变更后保留旧端点/参数约定部分时间（可选）
4. **文档化** — OpenAPI 中明确标注版本，旧版标记 deprecated

### 3.2 详细描述

- 现状统一为 v3 版本：`/api/v3/{module}/{resource}`
- API 变更分类：
  - **向下兼容**: 新增字段、可选参数、新增端点支持
  - **破坏性**: 删除字段/端点、参数名变更/必填变更、响应结构修改
- 破坏性变更政策：
  - 禁止在生产环境直接发布破坏性变更，需提前在文档中公告 30 天过渡期
  - 可采用内网分支先行验证，API 升级路径与产品同时发布
  - 提供旧端点兼容层（Spring Boot `@RequestMapping(value = "/oldPath", produces = "/v1")`）

### 3.3 交互流程

1. Propose 破坏性变更 → 在 API 文档中标注「Incompatible change」
2. 预公告 30 天给集成方通知
3. 部署时保留旧端点/逻辑（或标记 deprecated）
4. Beta 阶段向后兼容，GA 阶段正式切换

---

## 4. 非功能需求

- **性能要求**: 版本隔离不引入额外开销，DispatcherServlet 正常路由
- **安全要求**: 禁止在 API 中暴露内部实现逻辑
- **可用性要求**: 可通过 OpenAPI UI 查看当前 channel 的 v3 文档

---

## 5. 验收标准

- [ ] 所有 API 前缀统一为 `/api/v3/{module}/{resource}`
- [ ] OpenAPI 文档中标注每个端点的版本
- [ ] 破坏性变更仅在 v4（未来）规划，不破坏 v3
- [ ] 文档中明确「变更守则」与「公告模板」

---

## 6. 技术建议

### 6.1 涉及模块

- 后端 API 端点（统一前缀到 `/api/v3`）
- Swagger/OpenAPI 配置
- 内网健康检查（可选）对照

### 6.2 技术要点

- Spring Boot 通过 Controller 级别 `@RequestMapping(value = "/api/v3")` 统一前缀
- 破坏性变更从体系上规避：严格定义 openapi schema，尽可能使用可选字段/扩展字段
- 链路升级：新特性先在 `/api/v3/experimental` 展示（可选）

### 6.3 预估工作量

- 统一 API 前缀到 `/api/v3`：1 人天
- 定义变更规范与公告模式：1 人天
- 文档化 OpenAPI：<b>0.5 人天</b>

---

## 7. 依赖与风险

### 7.1 前置依赖

- 当前 API 版本前缀现状划分（从 issue 或 restful API 端点导入）

### 7.2 风险

- 破坏性变更不可避免时，集成方响应不及可能导致上线事故
- 版本策略与内部项目节奏对齐不足
