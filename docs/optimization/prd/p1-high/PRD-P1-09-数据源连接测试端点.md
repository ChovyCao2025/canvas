## 1. 问题描述

### 1.1 现状

- 数据源连接数配置后必须保存画布并执行 Dry-Run 才能验证凭据准确性
- 用户在保存前无法确认凭据/连接参数配置正确

### 1.2 痛点

- 凭据配置错误时无法提前发现，导致最终试结扫描后才报错
- 无法区分「参数语法错误」与「服务端凭据无效」

### 1.3 竞品对标

| 竞品 | 能力 |
|------|------|
| Braze | Connectors 可预连接验证 |
| Iterable | 活动验证 phone、email 联络点 |

---

## 2. 目标与价值

### 2.1 用户故事

> 作为 准备画布的用户，我希望 在保存前测试数据源连接是否有效，以便 修正凭据/参数并确保试旅无降级。

### 2.2 成功指标

- 测试端点响应时间 < 500ms
- 凭据错误场景提前识别率 > 90%

### 2.3 不做会怎样

- 无法提前校验连接，保存后才发现错误，返工成本高
- 无法区分具体错误类型，定位困难

---

## 3. 功能需求

### 3.1 核心功能

1. **连接测试 API** — POST `/api/v3/data-sources/{id}/test` 测试连接
2. **错误详情返回** — 区分「连接超时」「凭据无效」「权限不足」类型
3. **通过缓存优化** — 单日对同一连接多次测试不重复收费/调用

### 3.2 详细描述

- 请求：`{ "connectionConfig": "string" }`
- 响应：
  ```json
  {
    "success": true,
    "message": "Connected successfully",
    "latency": 150
  }
  ```
- 测试范围：
  - Database: 验证 JDBC 参数是否有效、是否可连接
  - MarketingPlatform: 验证 access_token 是否有效

---

## 4. 非功能需求

- **性能要求**: 连接测试响应 < 500ms（数据库类）；测试重试不超过 3 次
- **安全要求**: 不输出数据库内容，仅输出连接成功/失败

---

## 5. 验收标准

- [ ] 对有效凭据返回 success=true
- [ ] 对无效凭据返回详细错误信息
- [ ] 测试不影响生产环境连接池

---

## 6. 技术建议

### 6.1 涉及模块

- 后端 DataSourceController

### 6.2 技术要点

- 使用 HikariCP 测试连接或集成 WireMock 模拟外部服务

## 1. 问题描述

### 1.1 现状

- 全库 0 外键约束，数据完整性依赖应用逻辑
- 存在 orphan 数据（如 execution 属于不存在的 canvas）

### 1.2 痛点

- 应用逻辑侵入型检查容易遗漏场景
- 审计只能事后发现
- 无法在 E1 层面防止脏数据写入 + 孤儿对象关联

### 1.3 竞品对标

| 竞品 | 能力 |
|------|------|
| 生产级系统 | 强外键约束 |

---

## 2. 目标与价值

### 2.1 用户故事

> 作为 IT 运维，我希望 万一张量 MR 链中存在孤儿的执行踪迹，数据库能在事务层面直接拒绝，以便 精简在线上的 TCC 约束多用情况。

### 2.2 成功指标

- 恶意/非法数据在写入层面被拦截，不进入生产数据
- 数据可审计根因（谁、何时、什么场景）

---

## 3. 功能需求

### 3.1 核心功能

1. **外键表关系** — 配置核心 M:N 关系为 ON DELETE CASCADE
2. **写入校验** — MyBatis-Plus 事务拦截器在前置期检查外键冲突
3. **审计日志** — 任何外键约束冲突写入审计表

---

## 4. 非功能需求

- **性能要求**: 审计与外键链路不引入 10%+ MQ 延迟
