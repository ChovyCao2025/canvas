## 1. 问题描述

### 1.1 现状

当前所有列表页（画布列表、受众列表、节点列表、执行记录列表等）均无列排序功能。用户在查看大量数据时，无法按任意列排序，必须依赖浏览器搜索但不直观。

### 1.2 痛点

数据查找效率低下：用户需要反复翻页才能找到需要的数据；数据对比困难：无法按时间、优先级、状态等维度快速排序；数据洞察有限：无法快速识别异常数据（如异常高/常规低的数值）。

### 1.3 竞品对标

| 竞品 | 能力 |
|------|------|
| Excel | 支持多列排序、升序/降序切换、多级联合排序 |
| Salesforce | 自定义列排序、实时刷新、保存排序偏好 |

---

## 2. 目标与价值

### 2.1 用户故事

> 作为 运营人员，我希望 按任意列对数据进行排序列表，以便 快速找到目标数据、对比数据趋势、识别异常数据

### 2.2 成功指标

- 列表页支持至少 5 种常用列排序（时间、名称、状态、优先级、负责人）
- 排序状态通过 localStorage 持久化

### 2.3 不做会怎样

数据查找效率低下，用户必须手动翻页筛选，影响运营效率；数据对比困难，无法快速识别异常数据；用户体验落后于竞品，降低平台易用性评分。

---

## 3. 功能需求

### 3.1 核心功能

1. 列表列排序控件（升序 ▲/降序 ▼）
2. 按需支持多列联合排序（如：先按状态排序，再按创建时间排序）
3. 用户排序偏好持久化（localStorage 保存列顺序、排序方向）

### 3.2 详细描述

当前列表页使用 Ant Design Table 组件，需扩展 TableSorter 的功能。前端维护排序状态（columnKey、direction），调用后端 API 按排序参数加载数据。使用 Ant Design Table 的 `sorter` 属性和 `onSortChange` 回调实现。

#### 3.2.1 原生 Table 排序集成

为 Table 组件配置 sorter 函数，支持 `columnKey` 和 `order` 参数。示例：

```typescript
<Table
  columns={[
    { title: '画布名称', dataIndex: 'name', sorter: true },
    { title: '创建时间', dataIndex: 'createdAt', sorter: true },
  ]}
/>
```

### 3.3 交互流程

1. 用户进入列表页 → 默认按创建时间降序排列。2. 用户点击 '画布名称' 列标题 → 升序排列。3. 用户点击 '画布名称' 列标题再次 → 降序排列。4. 用户的排序偏好自动保存到 localStorage，刷新页面保留。

---

## 4. 非功能需求

- **性能要求**：排序响应时间 < 500ms（后端 API 分页查询优化）
- **安全要求**：无特殊安全要求（排序参数为纯字符串，无注入风险）
- **可用性要求**：支持 Tab 键导航到排序控件，支持 Enter 键切换排序方向

---

## 5. 验收标准

- [ ] 至少 3 个列表页支持列排序（画布列表、受众列表、执行记录列表）
- [ ] 支持升序/降序两种排序方向切换
- [ ] 排序状态通过 localStorage 持久化，刷新页面保留

---

## 6. 技术建议

### 6.1 涉及模块

- **前端模块**：`frontend/src/pages/canvas-list/`, `frontend/src/pages/audience-list/`, `frontend/src/pages/execution-record/`

### 6.2 技术要点

1. 使用 Ant Design Table 的 `sorter` 属性配置排序函数。
2. 前端维护 `sorting: { columnKey: string, direction: 'ascend' | 'descend' }` 状态。
3. 调用后端 API 时传递 `?sortColumn=name&sortDir=desc` 参数。
4. localStorage 保存 `canvas-list-sort: { columnKey: 'createdAt', direction: 'descend' }`。

### 6.3 预估工作量

- **开发**：3 人天
  - 1 人天：为 3 个列表页添加排序控件 + localStorage 持久化
- **测试**：2 人天
- **总计**：5 人天

---

## 7. 依赖与风险

### 7.1 前置依赖

- 无前置依赖，后端已支持分页查询 + 排序参数

### 7.2 风险

| 风险 | 级别 | 缓解措施 |
|------|------|----------|
| 后端未优化分页查询性能，大数据量（1000+ 条）排序时超时 | 中 | 后端对 `sortColumn` + `sortDir` 参数建立索引，测试 10000 条数据排序响应时间 < 1s |
| 多列联合排序逻辑复杂，联调成本高 | 低 | 先实现单列排序，MVP 阶段不支撑多列联合排序 |
