## 1. 问题描述

### 1.1 现状

当前权限仅支持 **4 个粗粒度角色**，无法区分同一角色下的细粒度权限：
- `ADMIN` - 超级管理员
- `OPERATOR` - 运营人员
- `VIEWER` - 浏览者
- `AUDITOR` - 审计员

权限范围的缺点：
- 无画布级权限：同一个画布，ADMIN 可编辑，VIEWER 只能查看
- 无节点级权限：同一画布的节点，某些节点只有特定角色可修改
- 无操作级权限：即使拥有画布权限，也不支持「只读/只分配/只测试」等细分操作

### 1.2 痛点

1. **协作困难**：团队成员无法精细控制画布编辑范围
2. **职责不清**：运营人员可以编辑不该管的部分（如系统配置节点）
3. **安全风险**：高权限角色泄露后，攻击者可轻易破坏画布或触发无限循环
4. **审计不足**：无法追踪「谁在什么时候改了哪个画布的哪个节点」

### 1.3 竞品对标

| 竞品 | 能力 |
|------|------|
| Braze | Canvas 级权限（创建/编辑/查看/删除节点）、角色独立授权 |
| Iterable | 资源级 RBAC，细粒度权限（读/写/执行/分享） |
| Marketo | 组织单元 + 角色映射，支持自定义权限模板 |

## 2. 目标与价值

### 2.1 用户故事

- **产品经理**：作为产品负责人，我希望为团队分配画布编辑/查看权限，以便区分不同技能角色的工作范围。

- **研发人员**：作为系统工程师，我希望通过节点级权限控制，限制对技术性节点（如 Groovy 脚本节点）的修改。

- **审计员**：作为合规审计人员，我希望查看画布的完整操作历史（谁在什么时间改了什么），以便满足审计要求。

### 2.2 成功指标

- 权限模型覆盖 3 层（画布级/节点级/操作级）
- 权限规则数量 > 20 种细分操作（如 `canvas:create`, `canvas:edit:node_123`, `node:execute:overall`）
- 权限变更立即生效，无需重启
- 权限审计日志覆盖率 100%

### 2.3 不做会怎样

- 团队成员权限模糊，协作时互相干扰
- 高权限操作无追溯，责任无法界定
- 无法满足企业级数据安全审计要求

## 3. 功能需求

### 3.1 核心功能

1. **RBAC 模型升级**：从「角色 → 权限」映射升级为「用户 → 资源 → 操作」的树状权限树
2. **画布级权限配置**：对每个画布配置允许可编辑用户/角色（权限包括：创建、编辑、查看、删除、分享、复制）
3. **节点级权限配置**：对每个节点配置角色级别的操作权限（编辑、删除、复制、引用）
4. **全局操作权限**：配置系统级操作权限（触发器管理、画布发布、模板管理）
5. **权限审计日志**：记录所有权限变更和操作记录（Who/When/What）

### 3.2 详细描述

#### 3.2.1 权限树结构

```
用户
└── 角色 (可选)
    └── 资源权限列表
        ├── 画布级权限 (canvas permissions)
        │   ├── 创建画布
        │   ├── 编辑画布 (编辑自己的画布)
        │   ├── 编辑画布 (编辑他人画布)
        │   ├── 查看画布
        │   ├── 删除画布
        │   ├── 分享画布
        │   └── 复制画布
        ├── 节点级权限 (node permissions)
        │   ├── 节点 ID: 123
        │   │   ├── 编辑节点
        │   │   ├── 删除节点
        │   │   ├── 复制节点
        │   │   └── 引用节点
        │   └── 节点 ID: 456
        │       ├── 编辑节点
        │       └── 查看节点配置
        └── 系统级操作 (system actions)
            ├── 管理触发器
            ├── 查看审计日志
            └── 配置系统参数
```

#### 3.2.2 权限检查注解

```java
public interface CanvasPermission {
    /**
     * 检查画布操作权限
     */
    @VaildCanvasPermission(action = "edit", resourceType = "canvas")
    String checkEditPermission(String canvasId, String userId);

    /**
     * 检查节点操作权限
     */
    @VaildNodePermission(action = "edit", nodeTypeKey = "groovy_script")
    void checkNodeEditPermission(Long nodeId, String userId);
}
```

实现逻辑：

```java
public class PermissionService {
    public boolean hasResourcePermission(String canvasId, String userId, String action) {
        // 1. 查询用户角色
        List<String> roleKeys = getUserRoles(userId);

        // 2. 查询画布配置的权限规则
        CanvasPermissionConfig config = canvasPermissionConfigMapper.selectByCanvasId(canvasId);

        // 3. 检查角色是否在允许列表中
        boolean hasPermission = config.getAllowedRoles().containsAny(roleKeys) ||
                               config.getOwnerRoles().contains(userId);

        // 4. 节点级权限覆盖画布级权限
        if (isNodeLevelAction(action)) {
            hasPermission = checkNodePermission(canvasId, nodeId, userId, action);
        }

        return hasPermission;
    }
}
```

#### 3.2.3 权限配置界面

```
画布权限设置
├── 基础设置
│   ├── 画布名称
│   ├── 所属空间
│   └── 所有权（我创建 / 团队创建）
├── 权限配置
│   ├── 角色权限映射
│   │   ├── [所有者]
│   │   ├── 研发团队 + [编辑画布] + [编辑节点]
│   │   ├── 运营团队 + [查看画布] + [复制画布]
│   │   └── 审计团队 + [查看画布] + [查看节点]
│   └── 用户权限设置（可选覆盖）
└── 节点权限
    ├── 123 (Groovy 节点) + [编辑节点]
    ├── 456 (API 调用节点) + [仅查看]
    └── 全局策略（默认允许所有人编辑）
```

### 3.3 交互流程

**权限配置流程**：

1. 画布拥有者点击「权限设置」按钮
2. 进入画布权限配置页面，选择「角色权限映射」标签
3. 点击「添加角色」，选择角色（如「研发团队」）
4. 选中该角色，勾选允许的操作（编辑画布、编辑节点）
5. 点击「节点权限」标签，为特定节点设置细分权限
6. 保存生效

**权限检查流程**（代码）：

1. 用户发起操作（如 `/canvas/123/edit`）
2. 后端 `PermissionInterceptor` 拦截请求
3. 查询用户角色 → 查询画布权限配置
4. 决策是否允许（画布级 OR 节点级）
5. 如不通过，返回 403 Forbidden，记录审计日志

## 4. 非功能需求

### 4.1 性能要求

- 权限检查延迟：< 10ms（用户操作响应时间影响 < 5%）
- 权限缓存：Redis 集中存储用户角色，TTL=1h，权限变更实时刷新

### 4.2 安全要求

- 权限校验必须后置（如执行权限检查后再解密请求参数）
- 权限模型支持「继承父级权限」（如画布级有「查看」权限，默认节点级也有「查看」权限）
- 所有权验证：用户无法修改他人拥有的画布的元数据

### 4.3 可用性要求

- 支持「贝叶斯权限」（即：加入团队后继承现有画布权限，避免逐个设置）
- 支持「权限模板」（如「开发画布模板」一键应用权限规则）

## 6. 技术建议

### 6.1 涉及模块

- **前端**：
  - `admin-frontend` - 权限配置 UI
  - `canvas-editor` - 节点级权限 UI（右键菜单「权限设置」）
- **后端**：
  - `canvas-api` - 权限校验接口（GET `/api/v1/canvas/{id}/permissions`）
  - `canvas-common` - 权限注解、拦截器
  - `canvas-engine` - 执行权限检查（节点执行前）

### 6.2 技术要点

1. **权限模型设计**：
   - 权限原子性：`CAN_EDIT` / `CAN_DELETE` / `CAN_EXECUTE` / `CAN_VIEW`
   - 资源所有权：`creator_id`, `owner_team_id`
2. **缓存策略**：
   - `` 中的权限规则缓存（TTL=1h）
   - 敏感操作（发布画布、删除执行记录）强制刷新缓存
3. **权限继承**：
   - 画布级权限默认下放至节点级（除非节点级显式限制）
   - 跨空间权限隔离（空间 A 的画布不可被空间 B 的角色访问）

## 7. 依赖与风险

### 7.1 前置依赖

- 需要确认现有 4 个角色的职责定位
- 需要梳理画布管理流程（创建 → 编辑 → 发布）

### 7.2 风险

1. **性能瓶颈**：权限检查可能在高并发下成为瓶颈
   - 缓解：Redis 分布式缓存 + 权限规则预加载
2. **审计复杂**：权限链路复杂时（角色继承），审计日志难以追溯
   - 缓解：记录「权限穿透链」（如：用户 A → 角色 B → 权限 C）
3. **向后兼容**：旧权限模型如何平滑迁移
   - 缓解：保留旧角色映射表，自动转换为权限树规则
