# 营销画布产品战略方案（2026-05-31）

> 4种逻辑合并为"双轨并行"方案：稳定轨60%（止血→闭环→升级→智能→自动）+ 创新轨40%（AI+私域），15项竞品实践 + 4层企微私域全量采纳，12个月5阶段推进

---

## 战略定位

**中国市场的差异化公式：Braze的能力 × 企业微信的渠道 = 没有人做过的组合**

- 国际竞品（Braze/Iterable/Klaviyo）有能力但没有企微渠道
- 国内竞品（神策/有赞/微盟）有企微但DAG引擎弱
- 我们：DAG引擎（已建成）+ 企微SCRM（新建）= 独家组合

---

## 双轨并行架构

```
轨道1（稳定轨 60%资源）：止血 → 效果闭环 → 运营升级 → 智能决策 → 自动化
轨道2（创新轨 40%资源）：AI写文案 → 企微私域 → AI建画布 → 预测决策 → AI自动化
```

两条轨道同步推进，轨道1保底线，轨道2出亮点。

## 阶段1落地决策（2026-05-31补充）

### 决策1：AI写文案技术路线 → **大模型+Schema约束**

| 路线 | 选择 | 理由 |
|------|------|------|
| A. 直接调大模型API | ❌ | prompt不好调、输出不稳定 |
| B. 大模型+Schema约束 | ✅ | 输出稳定可解析，2周可交付 |
| C. Fine-tune小模型 | ❌ | 需要训练数据，前期不值得 |

**实现方案**：
```
用户输入：目标（激活/转化/挽回）+ 受众特征（新客/老客/高价值）
    ↓
Prompt模板（内置3种风格：专业/活泼/紧迫）
    ↓
调用通义千问API（国内合规，延迟<2s）
    ↓
JSON Schema校验（title≤20字, body≤200字, cta≤10字）
    ↓
返回3版文案供选择
```

**工作量**：2周（1周后端API + 1周前端UI集成到消息节点）

### 决策3：归因轻量版事件模型 → **手动配置转化事件**

| 选择 | 理由 |
|------|------|
| A. 手动配置转化事件 | ✅ 2周可交付，够用，后续升级 |
| B. 接入事件SDK | ❌ 4周，需业务方配合埋点 |

**实现方案**：
```
画布发布时，运营配置：
  - 转化事件：order_completed / user_registered / user_activated
  - 归因窗口：7天（可调）

后端逻辑：
  1. 用户被画布触达 → 记录 canvas_id, user_id, timestamp 到 canvas_touchpoint 表
  2. 业务方调用 /api/v1/events 接口上报转化事件
  3. 归因服务：查找7天内该用户所有触达 → last-touch归因 → 更新画布收入统计
```

**工作量**：2周（1周后端API + 1周前端归因面板）

### 决策4：模板引擎 → **引入Liquid4J，L2起步直接兼容L4**

| 路线 | 选择 | 理由 |
|------|------|------|
| A. 自研模板引擎 | ❌ | 前期省2周，后期还2月；安全漏洞风险 |
| B. 引入Liquid4J | ✅ | L2-L4一步到位，安全沙箱已有，竞品兼容 |

**渐进暴露策略**：
```
阶段1(L2)：UI只暴露变量插入 + 条件逻辑
  运营看到：插入变量 ▼ [用户名/手机号/...]  |  条件 ▼ [如果/否则]
  底层生成：{{user.name}} {% if user.vip %}尊享{% else %}普通{% endif %}

阶段2(L3)：UI暴露循环 + 嵌套
  运营看到：商品列表 ▼ [最近浏览/购物车/...]
  底层生成：{% for item in user.recent_items %}{{item.name}}{% endfor %}

阶段3(L4)：高级模式，直接写Liquid
  运营看到：高级模式（切换开关）
  底层生成：完整Liquid，支持filter/custom tag
```

**选型**：Liquid4J（Java生态最成熟Liquid实现，MIT协议）
**关键优势**：Braze/Iterable/Klaviyo全用Liquid → 模板市场导入零成本

### 决策5：归因数据存储 → **MySQL按月分区 + 异步归档**

| 选择 | 理由 |
|------|------|
| A. MySQL单表 | ❌ 千万级查询慢 |
| B. ClickHouse | ❌ 多一套基础设施，现阶段太重 |
| C. MySQL按月分区+归档 | ✅ 零新基础设施，分区裁剪快，归档秒级 |

**实现方案**：
```sql
CREATE TABLE canvas_touchpoint (
    id BIGINT AUTO_INCREMENT,
    canvas_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    channel VARCHAR(32),
    touched_at DATETIME NOT NULL,
    event_type VARCHAR(64),  -- touch / convert
    revenue DECIMAL(12,2),
    PRIMARY KEY (id, touched_at)
) PARTITION BY RANGE (TO_DAYS(touched_at)) (
    PARTITION p202606 VALUES LESS THAN (TO_DAYS('2026-07-01')),
    PARTITION p202607 VALUES LESS THAN (TO_DAYS('2026-08-01')),
    ...
);
```

**容量估算**：100画布 × 10万用户 × 2次触达/月 = 2000万行/月，分区表完全扛得住
**归档**：`ALTER TABLE canvas_touchpoint DROP PARTITION p202501;`（秒级，不锁表）
**后续**：真到亿级时，分区表迁移ClickHouse是标准操作

### 决策6：AI能力 → **统一AI Gateway + 模型可插拔**

| 选择 | 理由 |
|------|------|
| A. 每个功能直接调API | ❌ 换模型改5处代码，无统一管控 |
| B. 统一AI Gateway | ✅ 换模型只改配置，限流/计费/审计/降级统一 |

**架构**：
```java
public interface AIService {
    Mono<AIResponse> generate(AIRequest request);
}

record AIRequest(
    AITaskType type,        // COPYWRITING / CANVAS_DESIGN / PREDICTION / ANOMALY_DETECT
    Map<String, Object> params,
    Optional<JsonSchema> outputSchema
) {}

record AIResponse(
    String content, String model, long latencyMs, double cost
) {}
```

**配置**：
```yaml
canvas:
  ai:
    provider: tongyi          # tongyi / wenxin / deepseek / mock
    rate-limit: 100/min
    timeout: 5s
    fallback: template        # 降级：template(预设模板) / error(报错)
    audit: true
```

**渐进**：

### 决策7：画布内嵌A/B → **分叉节点 + 折叠模式 + 抽屉配置（全都要）**

3种方案不是互斥，而是**不同场景**：

| 场景 | 方案 | 理由 |
|------|------|------|
| 2变体A/B | 分叉节点 | 直观，和DAG一致，运营一眼看懂 |
| 3+变体A/B/n | 折叠分叉 | 分支太多时折叠成聚合节点，点击展开 |
| 配置变体详情 | 抽屉面板 | 右键→抽屉，不占画布空间 |

**交互设计**：
```
运营操作流程：
  1. 右键任意节点 → "转为A/B测试"
  2. 抽屉面板配置：变体A(50%) / 变体B(50%) / +添加变体C
  3. 画布自动生成分叉：AB_SPLIT节点 → N条分支 → AB_MERGE汇合
  4. 变体>2时，分叉自动折叠成聚合节点（显示"3变体"徽章）
  5. 点击聚合节点 → 展开看全部分支
  6. 统计面板：每个变体的送达/打开/点击/转化 + 显著性检验
```

**技术实现**：
```
AB_SPLIT节点：
  - 输出routes=["variant_a", "variant_b", "variant_c"]
  - 流量分配：按百分比随机分流（seed=user_id保证同一用户始终进同一变体）
  - NodeResult.multiNext(routes, distribution)

AB_MERGE节点：
  - 等待所有分支完成（NodeGate barrier sync，已有实现）
  - 输出汇总统计

折叠状态：
  - 存在canvas_metadata.folded_ab_nodes
  - 前端渲染时：变体≤2展开，>2折叠
  - 折叠节点显示：图标 + "A/B/n (3变体)" + 展开按钮
```

**竞品验证**：
- Braze：分叉节点（Canvas Step → Experiment Path）
- Iterable：分叉+折叠（Split节点，多分支可折叠）
- Klaviyo：抽屉配置（A/B Test侧边栏）
- **我们全都要**：分叉(2变体) + 折叠(3+变体) + 抽屉(配置)

**工作量**：3周（1周后端AB_SPLIT/AB_MERGE + 1周前端折叠渲染 + 1周统计面板）

### 决策8：生命周期网格 → **RFM自动计算 + 运营可微调阈值（全都要）**

规则和RFM不是互斥，是**分层**：

| 层 | 数据来源 | 作用 |
|----|----------|------|
| 底层 | RFM自动计算 | 客观、自动、可复现 |
| 上层 | 运营微调阈值 | 灵活、业务适配 |

**RFM → 生命周期映射**：
```
R(最近购买天数)  F(频次)  M(金额)  →  生命周期
───────────────────────────────────────────────
<30天           高       高       →  Champion（冠军用户）
<30天           中       中       →  Loyal（忠诚用户）
<30天           低       低       →  New（新用户）
30-90天         中       中       →  At-Risk（风险用户）
>90天           低       低       →  Dormant（沉睡用户）
>180天          任意     任意     →  Churned（流失用户）

运营可微调：
  - 默认阈值：R<30=活跃, R<90=风险, R>90=沉睡
  - 运营可拖动滑块调整天数边界
  - 保存为"生命周期策略"，应用到所有画布
  - 支持多套策略（不同业务线不同阈值）
```

**数据流**：
```
每日定时任务（PowerJob/CanvasScheduleService）：
  1. 计算每个用户的R/F/M值（基于order事件 + customer_profile）
  2. 应用当前生命周期策略 → 生成lifecycle_stage标签
  3. 写入user_profile.lifecycle_stage字段
  4. 更新生命周期网格缓存（Caffeine L1 + Redis L2）
```

**生命周期网格UI**：

### 决策9：疲劳度优先级 → **三级可配置（全都要）**

3种方案不是互斥，是**3种模式，运营自选**：

| 模式 | 适用场景 | 做法 |
|------|----------|------|
| 先到先得 | 画布少、不冲突 | 谁先执行谁占配额（默认模式，零配置） |
| 画布优先级 | 重要画布不能被挤 | 运营给画布设P0-P3优先级，高优先级占配额 |
| 智能优先级 | 画布多、运营排不过来 | 基于历史效果（打开率×转化率）自动排序 |

**配置UI**：
```
┌─ 疲劳度策略 ──────────────────────────────┐
│                                             │
│  全局上限：每人每天 [3] 条 / 每周 [10] 条   │
│                                             │
│  冲突解决模式：                              │
│  ○ 先到先得（默认）                          │
│  ○ 画布优先级（运营手动排）                  │
│  ○ 智能优先级（基于历史效果自动排序）        │
│                                             │
│  优先级配置（画布优先级模式时显示）：        │
│  ┌──────────────────┬──────┐                │
│  │ 画布名称          │ 优先级│                │
│  │ VIP生日关怀       │ P0   │                │
│  │ 购物车挽回        │ P1   │                │
│  │ 周报推送          │ P2   │                │
│  └──────────────────┴──────┘                │
│                                             │
│  智能优先级权重（智能模式时显示）：          │
│  打开率权重 [0.4]  转化率权重 [0.6]         │
│  冷启动天数 [7] （无数据时用先到先得）       │
└─────────────────────────────────────────────┘
```

**执行逻辑**：
```
用户触达时：
  1. 查Redis：该用户今日已触达次数
  2. 如果未达上限 → 放行
  3. 如果已达上限 → 按当前模式处理：
     - 先到先得：直接跳过（记录fatigue_skip事件）
     - 画布优先级：比较当前画布优先级 vs 已触达画布最低优先级
       → 当前更高 → 替换最低优先级的触达（降级那条画布）
       → 当前更低 → 跳过
     - 智能优先级：比较当前画布效果分 vs 已触达画布最低效果分
       → 当前更高 → 替换最低效果分的触达
       → 当前更低 → 跳过
  4. 记录触达到Redis（canvas_id, channel, timestamp, priority/score）
```

**数据结构**（Redis Sorted Set）：
```
Key: fatigue:{tenant_id}:{user_id}:{date}
Value: Sorted Set
  - member: canvas_id:channel
  - score: 优先级分（P0=100, P1=75, P2=50, P3=25 / 或智能效果分0-100）
```

**竞品验证**：
- Braze：画布优先级（可拖拽排序）
- Iterable：先到先得 + 优先级覆盖
- Klaviyo：智能优先级（基于打开率）
- **我们全都要**：3种模式可切换

**工作量**：3周（1周后端疲劳度引擎+Redis结构 + 1周配置UI + 1周智能优先级算法）

### 决策10：控制组实现 → **画布级固定分桶 + 可配置比例（全都要）**

3种方案不是互斥，是**2个维度可配置**：

| 维度 | 选项 | 说明 |
|------|------|------|
| 分桶方式 | 随机 / 固定hash | 运营选择 |
| 作用范围 | 画布级 / 全局级 | 运营选择 |

**推荐默认**：画布级 + 固定hash（统计最严谨，且不影响其他画布）

**配置UI**：
```
┌─ 控制组配置 ──────────────────────────────┐
│                                             │
│  启用控制组 ☑                               │
│                                             │
│  控制组比例：[5] %                          │
│                                             │
│  分桶方式：                                  │
│  ○ 固定分桶（推荐）— 同一用户始终在同一组   │
│    Hash算法：user_id % 100 < 控制组比例     │
│  ○ 随机分桶 — 每次执行随机分配              │
│                                             │
│  作用范围：                                  │
│  ○ 画布级（推荐）— 仅本画布排除，其他画布   │
│    不受影响                                  │
│  ○ 全局级 — 控制组用户在所有画布中都不触达  │
│                                             │
│  ⚠️ 全局级控制组用户将永远收不到任何营销消息 │
└─────────────────────────────────────────────┘
```

**执行逻辑**：
```
画布执行时：
  1. 解析受众用户列表
  2. 对每个用户计算控制组归属：
     - 固定分桶：hash(user_id + canvas_id) % 10000 < control_ratio * 100
       → 同一用户+同一画布 → 始终同一结果（可复现）
       → 不同画布 → 不同结果（画布级隔离）
     - 随机分桶：ThreadLocalRandom.current().nextInt(100) < control_ratio
  3. 控制组用户：跳过所有发送节点，但记录触达事件（type=control）
  4. 归因面板：实验组转化率 vs 控制组自然转化率 → 增量效果
```

**归因面板展示**：
```
┌─ 增量效果 ────────────────────────────────┐
│                                             │
│  实验组（触达）：                            │
│    触达 10,000人 → 转化 320人 → 转化率 3.2% │
│                                             │
│  控制组（未触达）：                          │
│    人数 526人  → 转化 8人   → 转化率 1.5%   │
│                                             │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  增量转化率：+1.7%                           │
│  增量收入：  +¥17,000                        │
│  显著性：    p=0.003 ✅ 显著                 │
│  置信区间：  [+0.8%, +2.6%]                  │
└─────────────────────────────────────────────┘
```

**统计方法**：
- 比例检验：Fisher精确检验（样本小时）或Z检验（样本大时）
- 显著性阈值：p<0.05（可配置）
- 最小样本量提示：控制组<100人时显示"数据不足，结论不可靠"

**竞品验证**：
- Braze：画布级固定分桶（默认5%）
- Iterable：画布级 + 可调比例（1-50%）
- Klaviyo：全局级固定分桶
- **我们全都要**：画布级/全局级可选 + 固定/随机可选 + 比例可调

**工作量**：2周（1周后端分桶逻辑+归因计算 + 1周前端配置UI+增量面板）

```
  │                                                    │
```

### S1：人群运算（交集/并集/差集）

**竞品做法**：
- CleverTap：Segment Math — A∩B / A∪B / A-B，拖拽式组合
- Braze：Segment嵌套 — 在Segment内引用其他Segment
- Iterable：Segment组合 — 支持AND/OR/NOT逻辑

**方案：人群运算引擎 + 可视化组合器（全都要）**

```
人群运算引擎：
  1. 定义运算类型：INTERSECT(交集) / UNION(并集) / DIFF(差集/A-B)
  2. 支持嵌套：(A ∩ B) ∪ C — 递归解析运算树
  3. 计算方式：复用resolveUserIds()，对结果集做Set运算

  API设计：
    POST /api/v1/audience/compute
    Body: {
      "name": "高价值未流失",
      "operator": "INTERSECT",
      "operands": [
        { "type": "ref", "audienceId": 101 },     // 引用已有人群
        { "type": "ref", "audienceId": 102 },
        { "type": "computed",                      // 或内联规则
          "dataSourceType": "CDP_PROFILE",
          "ruleJson": "{...}" }
      ]
    }

  运算优化：
    - 小集合先算（减少内存）
    - 超过10万人群 → 走SQL层运算（不用resolveUserIds拉全量）
    - 运算结果可保存为新人群定义

可视化组合器：
  ┌────────────────────────────────────────────────┐
  │  人群组合器                                     │
  │                                                 │
  │  ┌──────────┐     ┌──────────┐                  │
  │  │ 高价值用户 │     │ 活跃用户  │                  │
  │  │ (12,340)  │  ∩  │ (45,600) │ = 8,920人       │
  │  └──────────┘     └──────────┘                  │
  │       │                                         │
  │       ∪  ┌──────────┐                           │
  │          │ 新注册用户│                           │
  │          │ (3,200)  │                           │
  │          └──────────┘                           │
  │       = 12,120人                                │
  │                                                 │
  │  运算：∩ 交集  ∪ 并集  - 差集  [+添加操作数]     │
  │  [保存为新人群]  [预览用户列表]                    │
  └────────────────────────────────────────────────┘
```

**SQL层运算优化**（10万+人群）：
```sql
-- 交集：INNER JOIN
SELECT a.user_id FROM audience_user_a a
INNER JOIN audience_user_b b ON a.user_id = b.user_id

-- 并集：UNION
SELECT user_id FROM audience_user_a
UNION
SELECT user_id FROM audience_user_b

-- 差集：LEFT JOIN + IS NULL
SELECT a.user_id FROM audience_user_a a
LEFT JOIN audience_user_b b ON a.user_id = b.user_id
WHERE b.user_id IS NULL
```

**工作量**：后端5天(运算引擎+API+SQL优化) + 前端5天(可视化组合器) | 阶段1

**灰度比例渐进策略**（做成配置项，运营可选模式）：

| 模式 | 策略 | 适用场景 |
|------|------|----------|
| 手动调整 | 运营手动拖动百分比 | 默认模式，运营掌控 |
| 自动渐进 | 1%→5%→10%→25%→50%→100%，每2小时自动升级（指标不劣化） | 无人值守场景 |
| 指标触发 | 转化率不低于当前版本时自动加5%，劣化则自动回滚 | 最激进，需足够样本量 |

配置：
```yaml

  canary:
    mode: manual              # manual / progressive / metric-triggered
    progressive-interval: 2h  # 自动渐进间隔
    progressive-steps: [1, 5, 10, 25, 50, 100]  # 渐进步伐
    metric-trigger:
      metric: conversion_rate # 监控指标
      threshold: -5%          # 劣化超过5%自动回滚
      min-sample: 500         # 最小样本量才判断
```
