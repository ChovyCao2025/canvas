## 关键依赖关系

```
阶段1（止血） ──必须完成──→ 阶段2（效果闭环）
    │                            │
    │                            ├── 归因是A/B统计的前提
    │                            ├── 归因是控制组的前提
    │                            └── 模板引擎是AI写文案的前提
    │
    └── 全局疲劳度是智能节流的前提

阶段2（效果闭环） ──数据积累──→ 阶段3（运营升级）
    │                            │
    │                            ├── 回执数据是STO训练的前提
    │                            ├── 归因数据是生命周期网格的前提
    │                            └── 模板引擎是AI建画布的前提

阶段3（运营升级） ──数据积累──→ 阶段4（智能决策）
    │                            │
    │                            ├── STO数据是预测模型的前提
    │                            └── A/B数据是多臂老虎机的前提
```

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 阶段1多租户改造影响面大 | 可能引入bug | 先在测试环境验证，分批迁移表（先canvas核心表，再CDP/audience表） |
| 阶段2归因需要业务事件数据 | 如果没有下单/注册等转化事件，归因算不出 | 先接最核心的1-2个转化事件（下单/注册），后续扩展 |
| 阶段3 AI写文案需要LLM API | 成本+延迟+合规 | 用国内大模型（通义/文心），缓存常见场景prompt，单次成本<0.01元 |
| 阶段3 STO需要足够历史数据 | 新用户无打开记录 | 冷启动用全局最优时间，积累7天数据后切换个人化 |
| 阶段4预测模型需要数据科学能力 | 团队可能缺ML工程师 | 先用规则模型（RFM+阈值），后续引入AutoML |
| 前端技术债（2084行编辑器） | 补功能越补越难维护 | 阶段1末尾安排2周重构：拆组件+引入Zustand，为阶段2-4铺路 |
