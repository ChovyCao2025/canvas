# Canvas 系统并发能力评估（老板汇报摘要版）

## 1. 一句话结论

> 当前系统并不是只能做到 500；500~1000 更像是“运维配套与治理”问题，真正到 2000+ 时，才需要进入平台级架构演进。

---

## 2. 当前能力怎么说最准确

- 当前系统**不能直接按配置里的 1000 并发对外承诺**
- 更合理的生产口径：
  - **4C / 8G 单实例**：80 ~ 150 稳定 execution 并发
  - **8C / 16G 单实例**：150 ~ 300 稳定 execution 并发
- **500 并发**：当前代码 + 合理运维配套 + 调优治理，可以实现
- **1000 并发**：当前代码基础上，在较强配套和一定代码治理下，有机会实现
- **2000+ 并发**：需要平台级架构演进

---

## 3. 代码能力 vs 运维配套能力

### 代码层边界
代码真正决定的问题包括：
- 主链路 Redis / MySQL 往返较多
- execution / request / quota 在线写较频繁
- 重任务和轻任务还没有完全隔离
- Redis 职责较重
- retry / replay / audience / groovy 共享资源较多

### 运维配套边界
不是代码缺陷，而是实施要求：
- MySQL 规格够不够
- Redis 是否高配或分层
- RocketMQ broker 是否足够
- 下游服务是否一起扩容
- 应用实例数是否合理
- 压测与监控是否完善

### 结论
- **500 ~ 1000**：主要还是配套资源和治理问题
- **2000+**：才是平台级架构问题

## 5. 并发目标对应的用户规模

> 并发不是客户数，而是高峰时刻同时触发执行的用户数。

| execution 并发目标 | 高峰同时触发用户数 | 估算 DAU（中性口径） | 估算中型客户数 | 说明 |
|---:|---:|---:|---:|---|
| 500 | 500 | 10万 ~ 30万 | 10 ~ 30 | 当前代码经调优和集群化后较现实目标 |
| 1000 | 1000 | 20万 ~ 50万+ | 20 ~ 50 | 需要明显治理 |
| 2000 | 2000 | 50万 ~ 100万+ | 50+ | 需要平台级演进 |

### 汇报口径
- **500 并发**：大致对应 10 万级到几十万级 DAU
- **1000 并发**：大致对应 20 万到 50 万以上 DAU
- **2000 并发**：大致对应 50 万到 100 万级以上 DAU

---

## 6. 500 / 1000 / 2000+ 三档目标

| 目标 execution 并发 | 当前代码判断 | 更准确的定性 |
|---:|---|---|
| 500 | 可行 | 当前代码 + 合理运维配套可实现 |
| 1000 | 有机会 | 当前代码基础上可争取实现 |
| 2000+ | 不建议直接承诺 | 需要平台级架构演进 |

---

## 7. 2000+ 目标下的架构设计方向

如果目标继续上提，重点不是继续堆单机，而是做这几件事：

1. **按任务类型拆执行通道**
   - 轻任务、标准任务、重任务分开

2. **把在线状态写入分层**
   - 强在线状态少量保留
   - 弱在线状态异步化
   - 分析/审计状态独立承接

3. **Redis 分层**
   - 执行态 Redis
   - 缓存/路由 Redis
   - bitmap / 大对象 Redis

4. **MQ 流量分治**
   - 正常流量与 retry 流量隔离

5. **下游依赖平台化治理**
   - 独立连接池
   - 熔断、限流、容量契约化

## 8. 2000+ 目标后续动作

### 建议新增一个专项 spec
聚焦：
- 目标定义
- 当前边界
- 架构演进方向
- 分阶段实施路径
- 验收标准

### 建议新增一个专项 plan
按 4 阶段拆：
1. 容量基线验证
2. 配套资源升级
3. 代码治理
4. 架构升级

---

## 9. 最终建议

1. 先把 500 和 1000 当作容量治理目标
2. 不要把当前阶段所有问题都归因于代码能力
3. 到 2000+ 时，再正式进入平台级架构演进立项
4. 报告与汇报口径都应区分：
   - 代码实现边界
   - 运维与配套实施边界
