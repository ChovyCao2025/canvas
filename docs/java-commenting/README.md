# Java Commenting Coordination Guide

Date: 2026-06-13

## Purpose

This guide defines how to coordinate a repository-wide Java commenting pass
without changing runtime behavior. It is designed for work split across
subagents by Maven module or source package.

The main goal should stay short. Put the full rules in this document and point
agents to it instead of pasting a long prompt into the goal.

## Goal Best Practice

A goal is best used as a compact objective and guardrail, not as the full
specification. Keep it short enough that every worker can remember the purpose,
then reference this guide for detailed rules.

Recommended goal:

```text
Add complete Chinese Javadocs and important inline comments to backend Java
sources without changing behavior. Follow docs/java-commenting/README.md and
dispatch work by module using docs/java-commenting/subagent-packets.md.
```

If the tool or workflow supports only a short goal, use:

```text
Document backend Java sources with Javadocs only; no logic changes. Follow
docs/java-commenting/README.md.
```

Do not put the entire commenting standard into the goal. Long goals are harder
to audit, easier to partially ignore, and harder to reuse across workers.

## Scope

Default scope:

- `backend/**/*.java`
- Exclude `target/`, `generated-sources/`, `build/`, `dist/`, and other
  generated output directories.
- Do not modify applied Flyway migrations.
- Do not modify Java logic, signatures, package names, imports, annotations, or
  formatting except where comment placement requires whitespace.

Module order:

1. `backend/canvas-cache-sdk`
2. `backend/canvas-engine`

Within `canvas-engine`, split by package or layer when possible:

- web/controller/API layer
- engine/node handler layer
- domain/application services
- persistence/data objects/mappers
- platform/config/infrastructure
- tests

## Commenting Rules

Every Java type must have a Javadoc comment:

- class
- interface
- enum
- annotation type
- nested type

Every field must have a Javadoc comment:

- `private`, `protected`, and `public` fields
- `static final` constants
- enum values
- injected dependencies
- configuration fields
- DTO fields

Every method must have a Javadoc comment:

- constructors
- public methods
- protected methods
- private methods
- interface methods
- overridden methods
- test helper methods

Important internal logic must have concise inline comments when the reason is
not obvious:

- complex branches
- state transitions
- transaction boundaries
- cache reads, writes, and invalidation
- database writes
- external system calls
- exception handling
- idempotency rules
- performance-sensitive logic
- concurrency or locking assumptions

All newly added Java comments must be written in Chinese. Keep Java identifiers,
annotation names, exception class names, method names, field names, and protocol
keywords unchanged.

## Record Conversion Rule

As an explicit exception to the comment-only scope, workers should convert Java
`record` types in their assigned scope to normal class declarations when they
touch those files.

This conversion is allowed only to improve readability for fully documented
fields. It must preserve the complete data contract:

- Keep every record component as a field. Do not drop, rename, merge, reorder,
  or reinterpret any component.
- Preserve component annotations on the corresponding fields, constructor
  parameters, or accessors as appropriate.
- Preserve the canonical constructor validation or normalization logic.
- Preserve accessor compatibility. If callers use record-style accessors such
  as `tenantId()`, keep equivalent methods unless all call sites are updated in
  the same assigned scope.
- Preserve `equals`, `hashCode`, and `toString` behavior. Prefer Lombok
  annotations such as `@Value` or `@Data` only when they match the previous
  record semantics and the project already uses Lombok in that style.
- Preserve serialization and framework behavior for Jackson, validation,
  MyBatis, Spring configuration binding, and tests.
- Add field-level Javadocs to every converted field and keep one blank line
  between documented fields.

If a record has non-trivial constructors, custom methods, annotations on
components, framework binding behavior, or external API exposure, treat the
conversion as risky. Convert only when compatibility can be preserved; otherwise
return `NEEDS_CONTEXT` with the exact file and reason.

## Dependency Injection Rule

Spring Boot dependency injection must follow constructor injection best
practices and should use Lombok where it improves readability. Avoid adding
multiple hand-written constructors just to make tests easier.

Required dependencies:

- Use `private final` fields.
- Prefer Lombok `@RequiredArgsConstructor` for Spring beans when no constructor
  parameter needs `@Value`, `@Qualifier`, `@Lazy`, or custom constructor logic.
- Do not add `@Autowired` to a single required-args constructor generated by
  Lombok.
- If an explicit constructor is still needed, keep exactly one injection
  constructor.

Configuration values:

- Prefer `@ConfigurationProperties` for grouped, reusable, or domain-specific
  configuration.
- Use `@Value` only for one or two local scalar values.
- If `@Value` is needed on constructor parameters, use one explicit constructor
  instead of Lombok for that class.

Optional dependencies:

- Prefer `ObjectProvider<T>` or `Optional<T>`.
- Avoid field injection and `@Autowired(required = false)` unless preserving
  legacy compatibility.
- Do not mix field injection, constructor injection, and setter injection for
  the same dependency group.

Testing:

- Do not add extra production constructors only for tests.
- Prefer mocks, `@TestConfiguration`, factory methods, or test fixtures.
- Replace test-only `Clock`, `ExecutorService`, `ObjectMapper`, or client
  overrides with injectable beans when practical.

Avoid these patterns:

- Multiple constructors on a Spring bean with one constructor marked
  `@Autowired` and the others kept only for tests.
- Constructor overload chains that only fill default dependencies such as
  `Clock.systemDefaultZone()` or `new ObjectMapper()`.
- Production code comments that justify extra constructors only for legacy unit
  tests.
- Field injection for required dependencies.

If a worker encounters a Spring bean with multiple constructors, it should
prefer consolidation to one injection path when compatibility is clear. If
consolidation might break tests, manual construction, or framework binding,
leave behavior unchanged and report the file under "needs human confirmation".

## Javadoc Format

Use standard Javadoc:

```java
/**
 * 用一句话说明业务用途。
 *
 * 可选补充规则、约束、副作用、单位、默认值、事务行为、线程安全或幂等性。
 */
```

Method Javadocs should use tags only when they add useful information:

```java
/**
 * 使用给定运行时上下文执行画布节点。
 *
 * @param context 包含输入数据和运行时元数据的执行上下文
 * @return 节点执行后产生的执行结果
 * @throws NodeExecutionException 节点执行失败时抛出
 */
NodeResult execute(NodeContext context);
```

Rules:

- Do not add `@return` for `void` methods.
- Do not add `@throws` for exceptions that are not declared or practically
  thrown by the method.
- Do not mechanically repeat parameter names.
- Document nullability, allowed values, units, and side effects when known.
- For `@Override` methods, use `{@inheritDoc}` only when the inherited contract
  is sufficient. Add implementation-specific details when behavior differs.

## Field Formatting

When every field has Javadoc, keep one blank line between field declarations:

```java
/**
 * 画布的唯一标识。
 */
private Long id;

/**
 * 展示给用户的画布名称。
 */
private String name;

/**
 * 画布当前的发布状态。
 */
private CanvasStatus status;
```

Do not collapse documented fields together:

```java
/**
 * 画布的唯一标识。
 */
private Long id;
/**
 * 展示给用户的画布名称。
 */
private String name;
```

## Readability Formatting

The commenting pass should make Java files easier to scan. Adding Javadocs to
every member can make a source file visually dense, so workers may adjust blank
lines around comments and members while keeping code behavior unchanged.

Use these spacing rules:

- Keep one blank line between package/import blocks and the type Javadoc.
- Keep one blank line between documented fields.
- Keep one blank line between field groups and constructors.
- Keep one blank line between constructors.
- Keep one blank line between methods.
- Keep one blank line before nested type Javadocs.
- Keep one blank line after a multi-line inline comment when it separates two
  logical steps.
- Do not add multiple consecutive blank lines unless the file already uses that
  style to separate major sections.
- Do not collapse a Javadoc block directly against the previous member.
- Do not run a whole-file formatter just to add comments.

Recommended class layout:

```java
/**
 * 协调画布执行请求。
 */
public class CanvasExecutionService {

    /**
     * 用于持久化执行状态的仓储。
     */
    private final ExecutionRepository executionRepository;

    /**
     * 用于避免重复查询执行状态的缓存。
     */
    private final ExecutionCache executionCache;

    /**
     * 使用必需的持久化和缓存依赖创建服务。
     *
     * @param executionRepository 用于持久化执行状态的仓储
     * @param executionCache 用于避免重复查询执行状态的缓存
     */
    public CanvasExecutionService(
            ExecutionRepository executionRepository,
            ExecutionCache executionCache) {
        this.executionRepository = executionRepository;
        this.executionCache = executionCache;
    }

    /**
     * 为请求中的画布启动一次新的执行。
     *
     * @param request 调用方提交的执行请求
     * @return 新建执行记录的标识
     */
    public Long startExecution(StartExecutionRequest request) {
        // 写入前先校验，避免无效请求产生部分状态。
        validate(request);

        return executionRepository.insert(request);
    }
}
```

## Inline Comment Format

Use `//` for one-line reasoning:

```java
// 事务提交后再失效缓存，避免回滚的数据变更误删有效缓存。
TransactionSynchronizationManager.registerSynchronization(...);
```

Use `/* ... */` for multi-line reasoning:

```java
/*
 * 只重试瞬时故障。业务校验错误必须快速失败，
 * 因为重试可能产生重复的外部副作用。
 */
if (isTransient(error)) {
    retry();
}
```

Inline comments should explain why the code exists, not translate the code.

## Quality Rules

Avoid low-value comments:

- `Gets user id.`
- `Sets name.`
- `Loops through list.`
- `Checks if value is null.`
- `Creates object.`

Prefer comments that explain business meaning and constraints:

- `拥有本次画布执行的租户标识。`
- `引擎将节点标记为失败前等待的超时时间，单位为毫秒。`
- `用于对 webhook 回调做幂等去重的外部事件键。`
- `使用数据库时间，确保不同应用节点上的过期判断保持一致。`

When the exact business meaning is unclear:

1. Write a conservative comment based only on the code.
2. Do not invent domain behavior.
3. Add the location to the final "Needs human confirmation" list.

## Behavior Guardrails

This task is documentation-only. Workers must not:

- change method bodies
- rename classes, methods, fields, variables, or packages
- add or remove annotations
- add or remove imports unless a Javadoc link introduces an import, which should
  generally be avoided by using plain text or fully qualified names
- reformat files unrelated to comment placement
- run whole-file formatters unless the coordinator explicitly requests it
- modify tests except to add comments
- modify generated code
- modify applied Flyway migrations

The only allowed structural exception is the `record` to normal class
conversion described in "Record Conversion Rule". That exception still must not
drop fields or change runtime behavior.

If a file has existing user edits, preserve them. Do not revert unrelated
changes.

## Verification

At the end of each module or worker packet:

1. Run a compile or focused test command when practical.
2. Confirm no non-comment behavior changes were introduced.
3. Report files changed and any locations requiring human confirmation.

Suggested commands:

```bash
cd backend
mvn -pl canvas-cache-sdk test
mvn -pl canvas-engine test
```

For very large batches, at minimum run:

```bash
cd backend
mvn -pl <module> -DskipTests compile
```

## Final Report Format

```text
status: DONE, DONE_WITH_CONCERNS, or BLOCKED
scope completed:
files changed:
verification run:
verification result:
needs human confirmation:
behavior changes: none
```
